# [웹개발] API 설계부터 실생활 예시까지, REST API 완벽 정복

안녕하세요! 웹 개발의 세계에 오신 것을 환영합니다. 개발자라면 누구나 'API'라는 단어를 매일 같이 듣게 되는데요. 막연히 '서버와 통신하는 거'라고 알고는 있지만, 정확히 무엇인지, 왜 중요한지, 어떻게 쓰이는지 궁금하셨을 겁니다.

오늘은 API의 기본 개념부터 우리 삶을 편리하게 해주는 실제 API 예시들, 그리고 웹 개발의 표준으로 자리 잡은 REST API의 핵심까지, A to Z로 완벽하게 정리해 보겠습니다.

## 1. API 설계, 대체 뭔가요?

API(Application Programming Interface)는 아주 간단히 말해 **프로그램과 프로그램이 서로 소통하기 위한 '메뉴판' 또는 '사용 설명서'**입니다.

웹 개발에서 API 설계란 "우리 서비스의 기능을 다른 프로그램이 어떻게 가져다 쓸 수 있는지에 대한 명확한 규칙을 만드는 과정"입니다.

API를 잘 설계하면 다음과 같은 엄청난 장점이 있습니다.

-   **개발 효율성 UP**: 프론트엔드와 백엔드 개발자가 약속된 API '메뉴판'을 보고 동시에 개발을 진행할 수 있습니다. 서로의 작업이 끝날 때까지 기다릴 필요가 없죠.
-   **높은 재사용성과 안정성**: 잘 만든 '사용자 정보 조회 API'는 웹, 모바일 앱, 관리자 페이지 등 여러 곳에서 재사용할 수 있습니다. 내부 로직이 바뀌더라도 API 규칙만 그대로라면, API를 사용하는 다른 서비스들은 아무런 영향을 받지 않습니다.

## 2. 우리 주변의 API 예시들 (feat. 카카오, 네이버)

"그래서 API가 실제로 어떻게 쓰이나요?"

API는 생각보다 우리 삶 아주 가까이에 있습니다. 우리가 매일 사용하는 서비스 대부분이 API를 기반으로 동작합니다.

### 예시 1: 소셜 로그인 API (카카오, 구글, 네이버 로그인)

> 웹사이트나 앱에서 "카카오로 로그인하기" 버튼을 눌러 회원가입이나 로그인을 대신하는 기능입니다.

개발자는 복잡하고 보안이 중요한 로그인 시스템을 직접 만들 필요 없이, 카카오나 구글이 **미리 만들어 놓은 강력한 로그인 기능(API)을 빌려 쓰는 것**입니다. 우리는 버튼 하나만 누르면 되지만, 그 뒤에서는 우리 앱과 카카오 서버가 API로 아주 바쁘게 소통하고 있답니다.

### 예시 2: 지도 API (네이버 지도, 카카오맵, Google Maps)

> 배달 앱에서 가게 위치를 보거나, 부동산 앱에서 매물 위치를 확인할 때 보이는 바로 그 지도 기능입니다.

배달 앱 회사가 전 세계 지도를 직접 만들 수는 없겠죠? 네이버나 구글이 **만들어 놓은 지도 데이터와 기능(API)을 빌려와** 내 서비스에 필요한 부분만 쏙 빼서 보여주는 것입니다.

### 예시 3: 결제 API (카카오페이, 토스페이먼츠, 아임포트)

> 온라인 쇼핑몰에서 상품을 결제하는 기능입니다.

우리가 [결제하기] 버튼을 누르는 순간, 쇼핑몰은 결제 전문 회사의 **결제 API를 호출**합니다. 이 API가 카드사와의 통신, 보안 처리 등 복잡한 모든 과정을 대신 처리하고 "결제 성공" 또는 "실패"라는 결과만 쇼핑몰에 알려줍니다.

### 예시 4: 공공 데이터 API (정부24, 공공데이터포털)

> 실시간 버스 도착 정보 앱, 미세먼지 농도 알림 앱 등이 바로 이 API를 활용합니다.

정부나 공공기관은 국민에게 유용한 데이터를 API 형태로 무료로 개방합니다. 개발자들은 이 **공공 API를 활용해** 교통, 날씨, 부동산, 보건 등 다양한 분야에서 새롭고 혁신적인 서비스를 만들어냅니다.

## 3. 그래서 REST API가 뭔데요?

이제 API가 무엇인지 감이 오셨을 겁니다. **REST(REpresentational State Transfer)**는 이런 API를 설계하는 여러 가지 스타일 중, 현재 웹에서 가장 널리 쓰이는 '국민 표준'입니다.

REST는 웹의 기존 기술(HTTP 프로토콜)을 그대로 활용하므로 이해하기 쉽고 직관적이라는 장점이 있습니다.

REST는 다음 세 가지 요소로 구성됩니다.

1.  **자원(Resource)**: API로 다루는 모든 데이터 (예: 사용자, 게시글)
2.  **행위(Verb)**: 자원에 대한 행동 (HTTP 메서드: GET, POST, PUT, DELETE 등)
3.  **표현(Representation)**: 자원을 나타내는 방식 (주로 JSON 형식)

이 세 가지를 조합하면 "어떤 자원(Resource)에 어떤 행위(Verb)를 할 것인지"를 명확하게 표현할 수 있습니다.

### REST API 실용 예시: 간단한 블로그

블로그의 '게시글(posts)'을 자원으로 다루는 REST API를 예시로 만들어 보겠습니다.

| 목적 | HTTP 메서드 | 주소 (URI) | 설명 |
| :--- | :--- | :--- | :--- |
| **모든 게시글 조회** | `GET` | `/posts` | 블로그의 모든 게시글 목록을 요청합니다. |
| **특정 게시글 조회** | `GET` | `/posts/1` | ID가 1인 게시글의 상세 정보를 요청합니다. |
| **새로운 게시글 작성** | `POST` | `/posts` | 새로운 게시글을 생성합니다. (내용은 본문에 담아 전송) |
| **특정 게시글 수정** | `PUT` | `/posts/1` | ID가 1인 게시글의 내용을 수정합니다. |
| **특정 게시글 삭제** | `DELETE`| `/posts/1` | ID가 1인 게시글을 삭제합니다. |

주소와 메서드만 봐도 어떤 기능을 할지 바로 예측이 가능하지 않나요? 이것이 바로 REST의 가장 큰 매력입니다.

## 4. [심화] REST의 핵심, '무상태성(Statelessness)' 파헤치기

REST API를 이야기할 때 절대 빼놓을 수 없는 개념이 바로 **무상태성(Statelessness)**입니다.

> **무상태성을 가장 쉽게 비유하면 '자판기'입니다.**
> 자판기는 방금 전 어떤 사람이 콜라를 뽑아 갔는지 기억하지 않습니다. 오직 현재 들어온 요청(돈+버튼)만으로 독립적인 작업을 완벽하게 수행하죠.

서버도 마찬가지입니다. **서버가 클라이언트의 이전 요청이나 상태를 전혀 기억하지 않는 것**이 바로 무상태성입니다. 모든 요청은 그 자체로 완결된, 독립적인 요청이어야 합니다.

-   **Stateful (상태 유지) 서버:** "아까 장바구니에 노트북 담으신 분 맞죠? 이제 결제해 드릴게요." (서버가 사용자를 기억해야 해서 힘듦)
-   **Stateless (무상태) 서버:** "['노트북']이 담긴 장바구니를 결제해달라는 요청이 왔군. 처리해야지." (요청에 모든 정보가 담겨 있어 서버는 기억할 필요가 없음)

무상태 구조에서는 서버가 사용자의 상태를 신경 쓸 필요가 없습니다. 덕분에 어떤 서버 컴퓨터가 요청을 받든 상관없이 동일한 작업을 처리할 수 있어 **서버를 여러 대로 늘리기(확장성)**에 매우 유리합니다.

## 마치며

오늘은 API의 기본 개념부터 우리가 매일 사용하는 서비스 속 API 예시, 그리고 웹 개발의 표준인 REST API와 그 핵심 특징인 무상태성까지 알아보았습니다.

API는 이제 웹과 모바일 서비스를 만드는 데 없어서는 안 될 핵심 요소입니다. 오늘 정리해 드린 내용이 여러분의 개발 여정에 훌륭한 나침반이 되기를 바랍니다.
