## CloudFormation (AWS SAA 시험 핵심 정리)

---

## 1. CloudFormation 개요

- **Infrastructure as Code (IaC)** 서비스
- JSON / YAML 템플릿으로 **AWS 리소스를 자동 생성·관리**
- 인프라를 코드로 관리 → **일관성, 재사용성, 자동화**

---

## 2. CloudFormation 핵심 개념 (시험 중요)

### 2.1 Template

- AWS 리소스를 정의하는 **설계도**
- 형식: **YAML (권장)** / JSON

---

### 2.2 Stack

- Template을 기반으로 생성된 **리소스 집합**
- Stack 단위로 **생성 / 업데이트 / 삭제**

**시험 포인트**
- Stack 삭제 시 **연결된 리소스도 함께 삭제**

---

## 3. Template 주요 구성 요소 (매우 중요)

- AWSTemplateFormatVersion
- Description
- Parameters
- Mappings
- Resources
- Outputs

---

### 3.1 Parameters

- 템플릿 실행 시 **외부 입력값**
- 환경별 값 분리 가능 (dev / prod)

**예시**
- EC2 InstanceType
- KeyName

**시험 포인트**
- 사용자 입력값 필요 → **Parameters**

---

### 3.2 Mappings

- **조건 없는 고정 매핑 테이블**
- 주로 **리전별 AMI ID** 설정

**예시**
- us-east-1 → ami-xxxx
- ap-northeast-2 → ami-yyyy

**시험 포인트**
- 리전별 값 분기 → **Mappings**

---

### 3.3 Resources ⭐ (가장 중요)

- 실제 AWS 리소스 정의 영역
- CloudFormation의 핵심

**예시**
- EC2
- S3
- RDS
- IAM Role

**시험 포인트**
- 모든 리소스는 **Resources**에 정의

---

### 3.4 Outputs

- Stack 생성 후 **결과값 출력**
- 다른 Stack에서 참조 가능

**예시**
- Load Balancer DNS
- VPC ID

**시험 포인트**
- Stack 간 연결 → **Outputs + Export**

---

## 4. Stack 동작 방식 (시험 단골)

### 4.1 Stack 생성

- Template 기반 리소스 자동 생성
- **의존성 자동 해결 (순서 자동 관리)**

---

### 4.2 Stack 업데이트

- 변경된 부분만 적용
- 필요 시 **리소스 교체(Replace)** 발생

**시험 포인트**
- EC2 InstanceType 변경 → **재생성 가능**

---

### 4.3 Stack 삭제

- Stack 삭제 시 **모든 리소스 삭제**
- 예외 처리 가능

**DeletionPolicy 사용**
- Retain
- Snapshot

---

## 5. 중요한 기능들 (시험 매우 중요)

### 5.1 Change Set

- Stack 변경 전 **변경 사항 미리보기**
- 실제 적용 전 검증 가능

**시험 포인트**
- 운영 환경 안전한 변경 → **Change Set**

---

### 5.2 Rollback

- 생성/업데이트 실패 시 **자동 원복**
- 실패 이전 상태로 복구

**시험 포인트**
- 배포 실패 자동 복구 → **Rollback**

---

### 5.3 DeletionPolicy

| 옵션 | 설명 |
|---|---|
| Delete | 기본값 |
| Retain | 리소스 유지 |
| Snapshot | 스냅샷 후 삭제 |

**시험 포인트**
- DB 보존 필요 → **Retain / Snapshot**

---

## 6. CloudFormation vs Elastic Beanstalk

| 항목 | CloudFormation | Elastic Beanstalk |
|---|---|---|
| 목적 | 인프라 자동화 | 애플리케이션 배포 |
| 제어 수준 | 매우 높음 | 낮음 |
| 대상 | 인프라 엔지니어 | 개발자 |

**시험 포인트**
- 인프라 자체 설계 → **CloudFormation**

---

## 7. 시험 시나리오 포인트

- 반복 인프라 배포 → CloudFormation
- 환경 일관성 유지 → CloudFormation
- 리전별 리소스 자동 구성 → Mappings
- 운영 중 변경 검증 → Change Set
- 배포 실패 자동 복구 → Rollback

---

## 8. 한 줄 요약

- **CloudFormation = 인프라를 코드로 관리**
- **시험 핵심 키워드**
    - Resources
    - Parameters
    - Change Set
    - Rollback  

---

## Amazon Pinpoint vs SES vs SNS 차이점 정리

---

### 서비스 목적 요약

| 서비스 | 핵심 목적 |
|---|---|
| Amazon Pinpoint | **대규모 사용자 대상 마케팅 캠페인** |
| Amazon SES | **이메일 전송 서비스** |
| Amazon SNS | **이벤트 기반 메시지 알림(Pub/Sub)** |

---

### 주요 기능 비교

| 항목 | Amazon Pinpoint | Amazon SES | Amazon SNS |
|---|---|---|---|
| 주요 용도 | 마케팅, 사용자 참여 | 트랜잭션 / 대량 이메일 | 알림, 이벤트 전파 |
| 메시지 유형 | Email, SMS, Push, In-App | Email | SMS, Email, HTTP, Lambda |
| 대상 관리 | 사용자 세그먼트 | 개별 이메일 주소 | 구독자(Subscriber) |
| 캠페인 관리 | ⭕ (스케줄, 타겟팅) | ❌ | ❌ |
| 사용자 행동 분석 | ⭕ | ❌ | ❌ |
| 메시지 자동화 | ⭕ (Journey) | ❌ | ⭕ (이벤트 트리거) |
| 실시간 이벤트 처리 | ❌ | ❌ | ⭕ |
| 마케팅 특화 | ⭕ | ❌ | ❌ |

---

### 사용 시나리오 비교 (시험 중요)

| 시나리오 | 적합한 서비스 |
|---|---|
| 신규 가입자 대상 환영 메시지 캠페인 | Amazon Pinpoint |
| 비밀번호 재설정 이메일 | Amazon SES |
| 주문 완료 알림 SMS | Amazon SNS |
| 수백만 사용자 대상 마케팅 푸시 | Amazon Pinpoint |
| 시스템 장애 알림 | Amazon SNS |
| 뉴스레터 발송 | Amazon SES |

---

### 시험에서 자주 나오는 키워드

| 키워드 | 연결 서비스 |
|---|---|
| 사용자 세그먼트 | Pinpoint |
| 마케팅 캠페인 | Pinpoint |
| 이메일 전송 | SES |
| 트랜잭션 이메일 | SES |
| Pub/Sub | SNS |
| 이벤트 기반 알림 | SNS |

---

### 한 줄 요약 (시험용)

- **Pinpoint** = 마케팅 + 사용자 분석 + 캠페인
- **SES** = 이메일 전송 전용
- **SNS** = 이벤트 알림 & Pub/Sub 메시징

## AWS Systems Manager (SSM) 정리

---

## 1. AWS Systems Manager 개요

- EC2 및 온프레미스 서버를 **중앙에서 관리**하는 AWS 서비스
- 서버 운영, 설정 관리, 자동화, 보안 관리 제공
- **에이전트(SSM Agent)** 기반 동작

---

## 2. SSM을 사용하는 이유 (시험 중요)

기존 서버 관리 문제
- 서버 개별 접속 필요
- 설정 변경 수동 작업
- 보안 취약 (포트 개방, Key 관리)
- 감사 및 추적 어려움

→ SSM으로 **중앙 집중 관리 + 자동화 + 보안 강화**

---

## 3. SSM 기본 동작 구조

1. EC2 인스턴스에 **SSM Agent** 실행
2. EC2에 **IAM Role** 연결
3. EC2가 AWS SSM Endpoint로 아웃바운드 통신
4. AWS Console / CLI에서 관리 수행

---

## 4. SSM 사용을 위한 필수 조건 (시험 필수)

### EC2 인스턴스 조건
- SSM Agent 설치
- 인터넷 또는 VPC Endpoint를 통한 AWS 접근 가능

### IAM Role 필수 정책
- `AmazonSSMManagedInstanceCore`

📌 시험 포인트
- SSM 동작 안 함 → **IAM Role부터 확인**

---

## 5. SSM 주요 기능 정리 (시험 단골)

---

### 5.1 Session Manager

- SSH / RDP 없이 EC2 접속
- 포트 개방 ❌
- Key Pair ❌
- IAM 기반 접근 제어 ⭕
- 세션 로그 기록 ⭕

---

### 5.2 Run Command

- 여러 EC2에 **명령어 일괄 실행**
- 서버 접속 없이 명령 수행
- 패치, 재시작, 설정 변경에 사용

---

### 5.3 Patch Manager

- OS 패치 자동 관리
- 패치 기준(Baseline) 설정 가능
- 정기 패치 스케줄링

---

### 5.4 Automation

- 운영 작업을 **워크플로우로 자동화**
- EC2 재시작, AMI 생성 등
- 실패 시 Rollback 가능

---

### 5.5 Parameter Store

- 설정 값 및 비밀 정보 저장
- 평문 / 암호화(SecureString) 지원
- KMS 연동

---

### 5.6 Inventory

- EC2 메타데이터 수집
- OS 정보, 설치 패키지, 설정 상태 파악

---

### 5.7 Maintenance Window

- 작업 수행 시간대 지정
- 업무 시간 외 작업 자동 실행

---

## 6. SSM vs 기존 관리 방식

| 항목 | SSM | 기존 방식 |
|---|---|---|
| 서버 접속 | 불필요 | 필요 |
| 포트 개방 | ❌ | ⭕ |
| 자동화 | ⭕ | ❌ |
| 중앙 관리 | ⭕ | ❌ |
| 감사 로그 | ⭕ | ❌ |

---

## 7. 시험 시나리오 포인트 (SAA)

- EC2를 프라이빗 서브넷에서 관리
- SSH 포트 개방 금지
- 다수 인스턴스에 명령 실행
- 보안 설정 값 중앙 관리
- 자동 패치 필요

→ **AWS Systems Manager**

---

## 8. 시험에 자주 나오는 키워드

- SSM Agent
- AmazonSSMManagedInstanceCore
- No inbound ports
- Run Command
- Session Manager
- Parameter Store

---

## 9. 주의 사항 (시험)

- SSM은 **애플리케이션 배포 도구 아님**
- EC2 또는 Managed Instance 대상 서비스
- IAM Role 없으면 동작 불가

---

## 10. 한 줄 요약

- **SSM = 서버를 접속 없이 안전하게 중앙 관리하는 AWS 표준 운영 도구**

## AWS Cost Explorer 정리

---

## 1. AWS Cost Explorer 개요

- AWS 사용 비용을 **시각적으로 분석**하는 비용 관리 서비스
- 과거 비용 분석 + 미래 비용 예측 제공
- 계정 및 서비스별 비용 추적 가능

---

## 2. Cost Explorer를 사용하는 이유 (시험 중요)

기존 비용 관리 문제
- 서비스별 비용 파악 어려움
- 비용 증가 원인 분석 어려움
- 예산 초과 위험

→ Cost Explorer로 **비용 가시성 확보 + 최적화**

---

## 3. Cost Explorer 핵심 기능

- 서비스별 비용 분석
- 계정 / 리전 / 태그 기준 비용 필터링
- 월별 / 일별 비용 조회
- **최대 12개월 비용 예측(Forecast)**
- Reserved Instance / Savings Plans 사용 현황 분석

---

## 4. 주요 분석 항목 (시험 단골)

| 분석 항목 | 설명 |
|---|---|
| Service | EC2, S3, RDS 등 서비스별 비용 |
| Usage Type | 데이터 전송, 스토리지, 컴퓨트 |
| Linked Account | 멀티 계정 비용 분석 |
| Tag | 프로젝트 / 팀별 비용 관리 |

---

## 5. Cost Explorer vs 기타 비용 관리 서비스

| 항목 | Cost Explorer | Budgets | Cost & Usage Report |
|---|---|---|---|
| 비용 시각화 | ⭕ | ❌ | ❌ |
| 비용 예측 | ⭕ | ❌ | ❌ |
| 알림 | ❌ | ⭕ | ❌ |
| 상세 데이터 | 중간 | 낮음 | 매우 높음 |
| 시험 출제 | ⭕ | ⭕ | ⭕ |

---

## 6. 시험 시나리오 포인트 (SAA)

- 서비스별 비용 증가 원인 분석
- 월별 비용 추이 확인
- 향후 비용 예측 필요
- 비용 최적화 대상 파악

→ **AWS Cost Explorer**

---

## 7. 주의 사항 (시험)

- Cost Explorer는 **기본 비활성화**
- 최초 사용 시 활성화 필요
- 데이터 반영까지 최대 24시간 소요

---

## 8. 시험에 자주 나오는 키워드

- Cost analysis
- Cost forecast
- Tag-based cost tracking
- Reserved Instance utilization
- Savings Plans

---

## 9. 한 줄 요약

- **Cost Explorer = AWS 비용을 분석하고 예측하는 시각화 도구**

## AWS Outposts 정리

---

## 1. AWS Outposts 개요

- AWS 인프라를 **온프레미스 데이터센터**에 확장하는 서비스
- AWS 리전과 **동일한 API, 도구, 서비스** 사용 가능
- 하이브리드 클라우드 구축을 위한 핵심 서비스

---

## 2. AWS Outposts를 사용하는 이유 (시험 중요)

기존 온프레미스 환경 문제
- 로컬 처리 지연(Latency)
- 규제 및 데이터 거주성 요구
- 클라우드 이전 어려움

→ Outposts로 **로컬 처리 + AWS 일관성 유지**

---

## 3. AWS Outposts 핵심 특징

- AWS가 관리하는 **물리적 랙(Rack) 장비**
- 고객 데이터센터에 설치
- AWS 콘솔에서 중앙 관리
- 리전과 **전용 네트워크 연결**

---

## 4. Outposts 제공 리소스

- EC2
- EBS
- RDS (제한적 지원)
- ECS / EKS (일부)

📌 시험 포인트
- 모든 AWS 서비스가 Outposts에서 동작 ❌

---

## 5. Outposts 동작 구조 (시험 단골)

1. AWS가 Outposts 랙을 고객사에 배송
2. 고객 데이터센터에 설치
3. AWS 리전과 **전용 연결** 구성
4. AWS 콘솔에서 리소스 생성 및 관리

---

## 6. Outposts vs 온프레미스 vs 리전

| 항목 | 온프레미스 | AWS Outposts | AWS 리전 |
|---|---|---|---|
| 위치 | 고객 데이터센터 | 고객 데이터센터 | AWS |
| 관리 주체 | 고객 | AWS | AWS |
| AWS API 사용 | ❌ | ⭕ | ⭕ |
| 지연 시간 | 매우 낮음 | 매우 낮음 | 높음 |
| 확장성 | 낮음 | 중간 | 매우 높음 |

---

## 7. 시험 시나리오 포인트 (SAA)

- 밀리초 단위 지연 시간 요구
- 데이터 로컬 저장 규제
- 기존 온프레미스 연계
- AWS와 동일한 운영 방식 필요

→ **AWS Outposts**

---

## 8. Outposts vs Local Zones vs Wavelength

| 서비스 | 목적 |
|---|---|
| Outposts | 온프레미스 AWS |
| Local Zones | 도심 저지연 |
| Wavelength | 5G 엣지 컴퓨팅 |

---

## 9. 주의 사항 (시험)

- Outposts는 **물리 장비 기반 서비스**
- 용량 확장은 추가 주문 필요
- 비용 높음
- 리전 연결 필수

---

## 10. 한 줄 요약

- **Outposts = AWS를 고객 데이터센터로 가져오는 하이브리드 클라우드**

## AWS Batch 정리

---

## 1. AWS Batch 개요

- 대규모 **배치 작업(batch job)** 실행을 위한 관리형 서비스
- 컴퓨팅 리소스 프로비저닝을 **자동으로 처리**
- 사용자는 **작업(Job)** 정의에만 집중

---

## 2. AWS Batch를 사용하는 이유 (시험 중요)

기존 배치 처리 문제
- 서버 용량 산정 어려움
- 작업 수에 따라 수동 스케일링 필요
- 비용 비효율

→ Batch로 **자동 스케일링 + 비용 최적화**

---

## 3. AWS Batch 핵심 개념 (시험 필수)

### Job
- 실행할 작업 단위
- 컨테이너 기반(Docker)

### Job Definition
- Job 실행 설정
- CPU, 메모리, 이미지, 명령어 정의

### Job Queue
- Job이 대기하는 큐
- 우선순위 기반 실행

### Compute Environment
- 실제 작업이 실행되는 컴퓨팅 리소스
- EC2 또는 Fargate 기반

---

## 4. AWS Batch 동작 방식 (시험 단골)

1. Job 제출
2. Job Queue에 적재
3. Batch가 필요한 컴퓨팅 리소스 자동 생성
4. Job 실행
5. 작업 종료 후 리소스 자동 정리

---

## 5. Compute Environment 종류

### Managed Compute Environment
- AWS가 인스턴스 생성/종료 자동 관리
- 가장 일반적인 사용 방식

### Unmanaged Compute Environment
- 사용자가 직접 EC2 관리
- 시험 출제 빈도 낮음

---

## 6. AWS Batch vs 기타 컴퓨트 서비스

| 항목 | AWS Batch | ECS | Lambda |
|---|---|---|---|
| 목적 | 배치 처리 | 컨테이너 서비스 | 이벤트 처리 |
| 실행 방식 | Job 기반 | Service / Task | 함수 |
| 실행 시간 | 제한 없음 | 제한 없음 | 최대 15분 |
| 스케일링 | 자동 | 설정 필요 | 자동 |
| 시험 출제 | ⭕ | ⭕ | ⭕ |

---

## 7. 시험 시나리오 포인트 (SAA)

- 대규모 데이터 처리
- 과학 계산 / 금융 모델링
- 비정기적 대량 작업
- 작업 수에 따라 컴퓨트 자동 확장 필요

→ **AWS Batch**

## 8. AWS Batch 사용 시 주의 사항 (시험)

- 실시간 처리 ❌
- 이벤트 기반 처리 ❌

- 컨테이너 기반 서비스

## 9. 시험에 자주 나오는 키워드

- Batch processing
- Job queue
- Compute environment
- Automatic scaling
- Cost optimization

## AWS Instance Scheduler 간단 정리

---

## 1. Instance Scheduler 개요

- **EC2 / RDS 인스턴스를 자동으로 시작·중지**하는 솔루션
- 사용하지 않는 시간대에 인스턴스를 꺼서 **비용 절감**

---

## 2. 주요 기능

- 일정(Schedule) 기반 자동 Start / Stop
- 태그(Tag)로 대상 리소스 지정
- EC2, RDS 지원
- 평일/주말, 업무 시간 등 스케줄 설정 가능

---

## 3. 동작 방식

1. 스케줄 정의 (예: 평일 9~18시)
2. 대상 인스턴스에 태그 추가
3. Lambda + EventBridge로 자동 제어
4. 지정 시간에 인스턴스 시작/중지

---

## 4. 언제 사용하는가 (시험 포인트)

- 개발/테스트 서버
- 업무 시간 외 미사용 인스턴스
- 비용 절감이 목표일 때

→ **Instance Scheduler**

---

## 5. 주의 사항

- 상태만 제어 (삭제 ❌)
- 항상 켜야 하는 프로덕션 서버에는 부적합
- AWS에서 제공하는 **솔루션(Solution)** 형태

---

## 6. 한 줄 요약

- **Instance Scheduler = 일정 기반으로 EC2/RDS를 껐다 켜서 비용을 줄이는 자동화 도구**
