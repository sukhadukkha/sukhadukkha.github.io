## OSI 7계층이란? 

![images](/assets/images/OSI.Png)


| 계층 | 이름 (Layer) | 데이터 단위 (PDU) | 주요 역할 | 관련 장비/프로토콜 |
|-----|--------------|------------------|----------|------------------|
| L7 | Application | Data | 사용자 인터페이스, 응용 서비스 | HTTP, FTP, DNS, ALB |
| L6 | Presentation | Data | 데이터 형식 변환, 암호화, 압축 | JPEG, MPEG, SSL/TLS |
| L5 | Session | Data | 통신 세션 유지 및 복구 | SSH, RPC, WinSock |
| L4 | Transport | Segment | 종단간 신뢰성 있는 전송 | TCP, UDP, 포트 번호 |
| L3 | Network | Packet | 최적의 경로 찾기(라우팅) | IP, ICMP, 라우터 |
| L2 | Data Link | Frame | 인접 노드 간 데이터 전송 | MAC 주소, 스위치 |
| L1 | Physical | Bits | 전기적, 물리적 신호 전송 | 케이블, 허브, 리피터 |


- `하위계층 : 데이터 이동의 물리적/논리적 기반(L1~L2)`
  - "같은 네트워크 안에서 옆집 컴퓨터와 어떻게 대화하는가"
  - L1(Physical): 데이터를 전기 신호(bit)로 바꿔 전선으로 보낸다.
  - L2(Data Link): MAC 주소를 사용해 인접한 장비 간에 데이터를 전달한다.

- `중위계층 : 어디든 찾아가는 경로(L3~L4)`
  - L3(Network): IP 주소를 기반으로 목적지까지의 최적의 경로(Routing)를 설정한다.
    - 패킷이 인터넷으로 갈지, 다른 서브넷으로 갈지 결정하는 동작
  - L4(Transport): 데이터의 신뢰성과 흐름 제어를 담당하며, Port 번호를 통해 어떤 프로세스에 전달할지 결정한다.
    - 보안 그룹 인바운드 규칙에 80번 포트를 추가하는 설정이 L4 제어다.

- `상위계층 : 애플리케이션과 데이터의 의미(L5~L7)`
  - TCP/IP 모델에서는 이 셋을 하나로 묶어 'Application' 계층으로 본다.
  - L5(Session): 통신하는 양 끝 단의 대화를 관리한다. (연결 유지, 끊기)
  - L6(Presentation): 데이터의 형식을 결정한다. (SSL/TLS, 압축, 인코딩)
  - L7(Application): 우리가 흔히 쓰는 서비스 프로토콜이다. (HTTP,DNS,FTP ...)

## 네트워크의 흐름 이해하기

- 데이터를 보낼 때는 7 -> 1계층으로 내려간다. 받을 때는 1 -> 7 계층으로 올라온다.
- 사용자가 친구에게 웹으로 "안녕" 이라는 메시지를 보내는 상황(보낼 때)
  - `내 컴퓨터 (7계층 -> 1계층: 캡슐화)`
  - `출발지 포트 번호는 Ephemeral Port 사용: 운영체제가 남는 포트 번호 중 하나를 무작위로 할당`  
  - L7: 브라우저가 HTTP 프로토콜을 사용해 Data를 생성한다.
  - L6: 메시지를 컴퓨터가 이해할 수 있게 인코딩하고, 암호화한다.
  - L5: 메시지를 보내기 위해 상대방 서버와 논리적인 연결 통로를 만든다.
  - L4: 데이터에 TCP 헤더를 붙인다. 여기에 출발지 포트 번호 + 목적지 포트 번호를 적는다. (세그먼트)
  - L3: 세그먼트에 IP 헤더를 붙인다. 친구 서버의 IP 주소를 적는다. (패킷)
  - L2: 패킷에 MAC 헤더를 붙인다. 우리 집 공유기나 다음 장비로 가기 위한 물리적 주소를 적는다. (프레임)
  - L1: 완성된 프레임을 0과1의 전기신호나 광신호로 바꿔서 밖으로 쏜다.
- `가는 중(중간 과정)`
  - 데이터가 인터넷 망을 타고 흘러가면서 중간중간 장비에서 L3계층 까지 까서 헤더를 확인해보며 목적지로 전달된다.
  - 데이터가 여러 라우터를 거쳐 먼 거리를 이동한다.
- `받을 때`
  - 친구 컴퓨터 (1계층 -> 7계층: 역캡슐화)
  - L1: 들어온 전기 신호를 다시 데이터(0,1)로 바꾼다.
  - L2: MAC 헤더를 뜯어본다. 나한테 온게 맞는지를 확인하고 헤더를 버린다. (프레임 -> 패킷)
  - L3: IP 헤더를 뜯어본다. 내 IP 주소가 맞는지를 확인하고 헤더를 버린다.(패킷 -> 세그먼트)
  - L4: TCP 헤더를 뜯어본다. 적힌 포트를 확인하고 헤더를 버린다. 또한 이 패킷이 순서대로 잘 왔는지, 빠진 건 없는지 확인하며 신뢰성을 보장한다.(세그먼트 -> 데이터)
  - L5 ~ L6: 암호화된 데이터를 복호화하고 세션을 종료할 준비를 한다.
  - L7: 최종 데이터인 "안녕"이 화면에 나타난다.

- `데이터 이동 상세 흐름`
  - 데이터가 서울에서 뉴욕으로 간다면?
  - Step 1: 내 컴퓨터는 3계층 헤더에 뉴욕 서버 IP를 적고, 2계층 헤더에는 우리집 공유기의 MAC주소를 적는다. 그리고 1계층(랜선)을 통해 신호를 쏜다.
  - Step 2: 중간에 있는 라우터는 L2를 까서 자기한테 온게 맞는지 확인한다. L3를 까서 최종 목적지 IP를 확인한다. "내 옆에있는 KT 라우터에 보내야 겠다" -> 다시 L2헤더를 만든다. 이때 출발지 MAC은 자기 자신, 목적지 MAC은 KT 라우터로 적는다. -> 광케이블을 통해 KT로 신호를 보낸다.
  - Step 3: 무한 반복, 이 과정이 해저 광케이블을 건너 뉴욕까지 수십 번 반복된다. 
- `그럼 이 과정에서 스위치는 어떤 역할일까?` 
  - 위의 과정에서 스위치를 추가해보자.
  - Step 1: 뉴욕IP로 보내야된다. 근데 일단 우리 집 라우터로 가야하니, 라우터 MAC 주소를 보낸다.
  - 스위치: 컴퓨터가 보낸 신호를 가장 먼저 받고, 라우터 MAC 주소로 가는 것을 확인하고, 그 라우터가 연결된 선으로만 데이터를 쏴준다.
  - Step 2: 수만은 라우터가 거대한 고성능 스위치에 붙어있다. 라우터가 "다음은 KT 라우터로 보내" 라고 한다면, 중간에 있는 스위치가 그 KT 라우터가 꽂힌 정확한 구멍을 찾아 전달해준다.
  - Step 3: 뉴욕에 도착한 후, 뉴욕 라우터는 IP를 확인하고 도착지가 맞는 것을 확인한다.
  - 스위치: 우리 네트워크 안에 정확히 어떤 MAC 주소로 가야하는지를 확인하고 서버로 데이터를 전달한다.
- `정리한 구조`
  - 스위치 -> 라우터 -> MAC 변경 -> 선 -> 스위치 -> MAC 주소 확인 -> 라우터 
- `나눠놓은 이유는?`
  - 만약 중간중간에 목적지만을 확인하지 않고 전달되는 프레임의 헤더를 하나씩 다 까서 4계층 or 7계층 까지 까본다면 인터넷 속도는 엄청나게 느려질 것이다.
  - 나눠놓고 역할 분담을 함으로써 효율적인 데이터 전달이 가능하다.

## TCP/ IP의 차이점

- IP는 목적지 호스트까지 패킷을 전달하는 `경로`를 책임지고, TCP는 그 호스트 안의 `특정 프로세스(Port)를 찾아` 데이터를 안전하고 순서대로 전달하는 `연결 신뢰성`을 책임진다.
- 클라우드에서의 통신 제어
  - IP(L3) 제어: 특정 국가나 IP 대역의 접속을 막고싶다 -> NACL이나 라우팅 테이블에서 제어
  - TCP(L4) 제어: 웹 서비스(80,443)만 열고 DB(3306) 접속은 막고싶다 -> 보안 그룹에서 포트 번호로 제어
