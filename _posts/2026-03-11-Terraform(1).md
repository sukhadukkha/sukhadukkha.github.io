## Terraform이 무엇인가?

- 오픈소스 코드형 IaC(Infrastructure as Code) 도구
- 가상 머신, 네트워크, 저장소 같은 인프라 자원을 GUI가 아닌 선언적인 코드로 정의하고 관리하게 해주는 도구다.


## Terraform을 왜 사용할까?

- 인프라 변경 사항을 Git 같은 버전 관리 시스템에 기록하고, 언제, 누가, 무엇을 바꿨는지 이력 추적이 가능하고, 롤백하기 쉽다.
- 콘솔에서 일일이 클릭하는 대신 코드로 실행하므로 반복 작업이 자동화되고, 실수를 예방하고 인프라 구축 속도가 빨라진다.
- 코드를 몇 줄 수정하는 것만으로 인프라 자원을 늘리거나 줄일 수 있다. 변화하는 요구사항에 맞춰 유연하게 인프라 규모를 조정할 수 있다.
- 인프라 설정을 파일로 공유하므로, 애플리케이션 코드를 짜는 듯이 협업이 가능하다.
- 개발,테스트,운영 환경에 동일한 인프라를 반복해서 생성할 수 있다. 수동 설정 시 발생하는 실수와 환경 간 차이를 없애준다.
- 클라우드 콘솔에 들어가 보지 않아도 코드 파일만 보면 현재 인프라 상태를 알 수 있다.
- 다양한 프로바이더를 HCL 언어(하나의 언어)로 관리할 수 있다.


## HCL

- 테라폼을 사용하기 위한 언어
- 선언적 언어 : 최종적인 상태를 작성한다.


## Terraform Core, Providers, Resources, State, Modules

- `Terraform Core`
  - 작성한 설정 파일을 읽고 리소스 간의 의존성 그래프를 생성한다.
  - 현재 상태와 코드를 비교하여 무엇을 추가, 수정, 삭제할지 결정한다.(plan)
- `Providers`
  - 테라폼이 외부 서비스와 통신하기 위한 플러그인이다.
  - Core가 요청하면 AWS Provider가 이를 AWS API 호출로 번역해서 요청을 실행한다.
- `Resources`
  - 테라폼으로 관리하는 최소 단위의 인프라 객체다.
  - VPC, EC2 Instance, S3 등이 모두 리소스이다.
- `State`
  - 테라폼이 관리하는 인프라의 현재 상태를 기록한 지도(terraform.tfstate 파일)다.
  - Source of Truth: 실제 클라우드에 배포된 자원과 내가 작성한 코드를 연결해주는 매핑 파일이다. 이 파일이 유실되면 테라폼은 내가 만든 인프라가 무엇인지 알 수 없게 되므로 매우 중요하다.
- `Modules`
  - 관련된 리소스들을 하나로 묶은 재사용 가능한 패키지다.
  - ex) VPC + EC2 + DB를 모듈로 만들어두면 프로젝트마다 코드를 복사붙여넣기 할 필요 없이 모듈 호출만으로 같은 환경을 구성할 수 있다.


## Terraform Init

- 현재 디렉토리를 테라폼 작업 디렉토리로 초기화하고 다음 세가지를 수행한다.
- `Provider 플러그인 다운로드`
  - 코드를 읽고 aws, google, github 등 어떤 라이브러리가 필요한지 파악한다. 그 후 하시코프 레지스트리에서 해당 플러그인을 다운로드하여 `.terraform` 폴더에 저장한다.
  - 프로바이더 버전이 바뀌면 다시 init을 해야 할 수도 있다.
- `백엔드 초기화`
  - 테라폼의 상태 파일인 `State`를 어디에 저장할지 결정한다. 로컬 PC OR AWS S3 같은 원격 저장소에 저장할지 설정된 경로를 확인하고 연결을 준비한다.
- `Module 다운로드`
  - 코드 내에 외부 모듈을 사용했다면, 해당 모듈의 소스코드를 가져와서 로컬에 복사본을 만든다.
- `실행 후 생성되는 중요한 파일들`
  - `.terraform/` 폴더: 다운로드한 프로바이더 플러그인과 모듈이 담기는 숨김 폴더이다.(용량 크므로 보통 Git에는 올리지 않음)
  - `.terraform.lock.hcl`(Dependency Lock File): 사용중인 프로바이더의 정확한 버전과 체크섬을 기록한다.
    - 팀원들과 협업할 때 모두가 동일한 버전의 플러그인을 사용하도록 보장하는 안전장치다.

## Terraform Plan

- 실제 인프라를 변경하지 않고, 코드를 실행했을 때 "어떤 자원이 생성, 수정, 삭제될지" 예측해서 보여주는 명령어다.
- `동작 원리(3-Way Check)`
  - plan을 수행할 때 다음 세 가지를 대조한다.
  - 현재 코드(HCL): 내가 작성한 최신 설계도
  - 상태 파일(State): 테라폼이 기억하고 있는 마지막 인프라 모습
  - 실제 인프라(Actual): 클라우드에 실제로 존재하고 있는 자원(Refresh 과정)
- `결과 읽는 법 (기호의 의미)`
  - `+ create`: 새로운 리소스 생성
  - `~ update-in-place`: 기존 리소스 삭제하지 않고 설정만 변경
  - `- destroy`: 기존 리소스 삭제
  - `-/+ replace`: 리소스를 삭제한 후 다시 생성 (수정만으로는 변경할 수 없는 속성을 바꿀 때 발생, 서비스 중단이 생길 수 있어 주의 필요)
- `왜 중요한가?`
  - 실수 방지
  - 협업 및 코드 리뷰
  - 예측 가능성
- `terraform plan -out=tfplan`: 계획 결과를 파일로 저장한다. 저장된 파일을 `terraform apply tfplan`으로 실행하면 plan 시점과 apply 시점 사이의 인프라 변경으로 인한 사고를 방지할 수 있다.

### Resource Graph

- 어떤 리소스를 어떤 순서로 처리할지 결정하기 위해 내부적으로 그리는 그래프이다.
- 효율적이고 정확한 배포를 가능하게 해주는 중요한 개념이다.
- `핵심 역할`
  - `의존성 파악 (Dependency Mapping)`: 인프라 자원들은 서로 연결되어있다. 보안 그룹이 먼저 만들어져야 EC2에 적용할 수 있다. 테라폼은 코드를 분석해 이런 선후 관계를 파악 후 그래프 형태로 연결한다.
  - `병렬 처리(Parallelism)`: 그래프를 그려보니 서로 아무런 연관이 없는 리소스들이 있다면 테라폼은 이를 동시에 생성함으로써, 전체 배포 시간을 획기적으로 단축한다.
    - 10개씩 병렬처리가 기본, -parallelism= 명령 통해 조절 가능
    - 언제 늘릴까? 관리해야 할 리소스가 수천 개이고, 의존성이 별로 없어서 배포 시간 단축해야 할 때
    - 언제 줄일까? API 요청 제한이 깐깐한 환경이거나, 네트워크 대역폭이 좁을때
- 두 가지 의존성 유형
  - `암시적 의존성(Implicit Dependency)`: 코드를 통해 자연스럽게 연결되는 경우
    - 한 리소스의 속성을 다른 리소스가 참조하면, 테라폼은 자동으로 판단한다.
  - `명시적 의존성(Explicit Dependency)`: 코드상으로 연결은 없지만 순서가 강제되어야 할 때 사용한다. (depends_on 사용)


## Terraform Apply

- 이 명령어는 코드를 바탕으로 실제 환경에 리소스를 생성, 수정, 삭제하라는 최종 승인 명령어이다.
- 작동 과정
  - `내부적 plan 실행`: 만약 미리 생성한 plan 파일이 없다면, 다시 한번 plan을 수행하여 변경 사항을 보여준다.
  - `사용자 확인`: "정말 실행할까요?"라는 메시지와 함께 yes 입력을 기다린다. (이때 아니면 못 멈춤)
  - `리소스 생성/변경`: 리소스 그래프에 따라 설정된 병렬 처리(기본 10개)를 활용해 실제 API 요청을 보낸다.
  - `State 파일 업데이트`: 작업이 끝나면 실제 인프라의 상태를 terraform.tfstate 파일에 기록한다.
- `자주 사용하는 옵션`
  - `-auto-approve`: 확인 단계 건너뛰고 바로 실행 / CI/CD 파이프라인에서는 사람이 직접 yes를 칠 수 없기에 필수적으로 사용
  - `terraform apply "plan_file`: 저장해둔 계획 파일을 그대로 실행
- Apply 완료 후의 변화
  - Apply가 성공하면 디렉토리에 `terraform.tfstate` 파일이 생성되거나 업데이트 된다.
- `Apply 도중 리소스 생성이 실패하면?`
  - 성공한 리소스: 에러가 나기 전까지 성공적으로 생성된 리소스는 클라우드에 그대로 남는다.
  - 실패한 리소스: 에러가 발생한 리소스에서 작업이 중단된다.
  - 남은 리소스: 실패한 리소스 이후에 만들어질 예정이었던 리소스들은 아예 시도조차 하지 않는다.
  - `State 파일은 실시간으로 업데이트된다`: 테라폼은 성공한 부분까지는 즉시 `terraform.tfstate` 파일에 기록한다.
- 해결법: 수정 후 다시 Apply (Idempotency) 
  - 에러 분석: 출력된 에러 메시지를 확인 (권한부족, 오타, 쿼터 제한 등)
  - HCL 수정: 에러 원인을 수정한다.
  - 다시 apply: 수정 후 다시 terraform apply를 실행한다.

## Terraform Destroy

- 테라폼이 State에 기록된 모든 자원을 클라우드상에서 완전히 제거하는 명령이다.
- 작동 원리
  - 생성할 때와는 정반대의 순서로 리소스를 삭제한다.
  - 생성 시: VPC -> Subnet -> EC2
  - 삭제 시: EC2 -> Subnet -> VPC
- 실행 과정
  - `Destruction Plan`: 어떤 자원이 삭제될지를 보여준다. (- 기호)
  - `사용자 확인`: 사용자에게 yes 입력 받기를 기다린다.
  - `삭제 수행`: 클라우드 API를 호출해 자원을 제거한다.
  - `State 업데이트`: 삭제가 완료되면 `terraform.tfstate` 파일에서 해당 리소스 정보를 제거한다.
- `주요 옵션 및 주의사항`
  - `-auto-approve`: 확인 절차 없이 즉시 삭제한다. CI/CD 환경이나 테스트 환경을 통째로 날릴 때 유용하다.
  - `-target`: `terraform destory -target=aws_instance.web` 처럼 특정 리소스만 골라서 삭제할 수 있다. ((하지만 의존성 문제가 생길 수 있어 실무에선 주의해서 써야 함)
  - `복구 불가능`: 이 명령어를 실행하고 나면 클라우드 상의 데이터와 설정은 완전히 사라지고 복구가 불가능하다.

## Terraform Validate

- 현재 디렉토리에 있는 테라폼 설정 파일들이 기술적으로 올바른지 확인한다. 
- `클라우드 API를 호출하지 않고 로컬에서만 수행된다.`
- 검토하는 것
  - `구문 오류(Syntax Errors)`: 괄호를 안 닫았거나 오타가 있는지
  - `속성 이름 확인`: 리소스에 존재하지 않는 속성을 사용했는지
  - `타입 체크`: 숫자가 들어가야 할 곳에 문자열이 들어가지는 않았는지
  - `필수 인수 누락`: 반드시 넣어야 하는 설정(S3 버킷의 이름 등)이 빠졌는지
  - `참조 오류`: 존재하지 않는 변수나 리소스를 참조하고 있는지
- `실행 시점과 조건`
  - `실행 시점`: `terraform init` 다음에 실행한다. (초기화가 되어야 프로바이더의 규칙을 알 수 있기 때문)
  - 특징: 인터넷 연결이나 클라우드 인증이 없어도 실행 가능하다.

## 정석 워크플로우

- `Write`: HCL로 .tf 파일 작성
- `Init`: terraform init 실행 (플러그인 및 모듈 다운로드, 백엔드 설정)
- `Validate/Fmt`: 코드 문법 검사 및 스타일 정리
- `Plan`: 실행 계획 확인
- `Apply`: 실제 인프라 생성
- 만약 코드를 짜다가 새로운 프로바이더를 추가(AWS만 사용하다가 Github 리소스도 관리하고 싶을 때 등)했다면 다시 한번 `terraform init`을 실행해줘여한다.
