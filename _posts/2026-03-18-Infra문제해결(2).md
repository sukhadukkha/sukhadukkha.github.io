## 상황

새로운 버전의 웹 애플리케이션 파드를 배포하기 위해 매니페스트를 적용했지만, 파드가 Running 상태로 넘어가지 못하고 계속 대기 중입니다. kubectl get pods를 확인해 보니 상태가 ImagePullBackOff로 표시됩니다.<br>
제공된 파드 설정과 이벤트를 분석하여 배포가 실패한 원인을 찾고 해결하세요.

- 제공 데이터들

```
kubectl get pods

NAME                             READY   STATUS             RESTARTS   AGE
web-frontend-7f8d9b4c5d-x2j4k    0/1     ImagePullBackOff   0          5m
--------------------------------------------------------------------------------

web-frontend-deployment.yml(일부)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-frontend
spec:
  replicas: 1
  template:
    spec:
      containers:
      - name: frontend
        image: nginx:1.25.12-alpine
        ports:
        - containerPort: 80
--------------------------------------------------------------------------------
kubectl describe pod web-frontend-7f8d9b4c5d-x2j4k

Name:         web-frontend-7f8d9b4c5d-x2j4k
Namespace:    default
Status:       Pending
...
Events:
  Type     Reason     Age                From               Message
  ----     ------     ----               ----               -------
  Normal   Scheduled  5m                 default-scheduler  Successfully assigned default/web-frontend-7f8d9b4c5d-x2j4k to node-01
  Normal   Pulling    4m (x3 over 5m)    kubelet            Pulling image "nginx:1.25.12-alpine"
  Warning  Failed     4m (x3 over 5m)    kubelet            Failed to pull image "nginx:1.25.12-alpine": rpc error: code = NotFound desc = failed to pull and unpack image "docker.io/library/nginx:1.25.12-alpine": failed to resolve reference "docker.io/library/nginx:1.25.12-alpine": docker.io/library/nginx:1.25.12-alpine: not found
  Warning  Failed     4m (x3 over 5m)    kubelet            Error: ErrImagePull
  Normal   BackOff    3m (x4 over 4m)    kubelet            Back-off pulling image "nginx:1.25.12-alpine"
  Warning  Failed     3m (x4 over 4m)    kubelet            Error: ImagePullBackOff
```

- `kubectl describe pod`: 특정 Pod의 상세 정보를 사람이 읽기 쉬운 형태로 출력하는 명령어 (Pod 자체의 메타데이터/상태/이벤트 (Kubernetes 레벨))
  - 언제 사용?
  - Pod가 Pending 상태일 때 (노드 자원 부족, 스케줄링 실패 원인 확인)
  - CrashLoopBackOff일 때 (컨테이너가 왜 죽는지 이벤트 확인)
  - Pod이 뜨긴 했는데 이상할 때 (환경변수, 볼륨 마운트 제대로 됐는지 확인)
  - `kubectl logs`: 컨테이너 내부 애플리케이션 로그(앱 레벨)

- Pod에 문제가 생기면 describe로 Events 먼저 확인 후 logs로 앱 로그 보는 순서로 디버깅한다.


## 해결

- Events 로그에서 `docker.io/library/nginx:1.25.12-alpine: not found` 로그를 확인할 수 있다.
- 이는 Docker Hub에 해당 이름 및 태그를 가진 이미지가 존재하지 않는다는 것을 의미한다.

- 해결 방법
  - Deployment 파일에서 컨테이너의 이미지를 존재하는 버전으로 수정한다.
  - 수정된 파일을 다시 적용한다 (`kubectl apply -f web-frontend-deployment.yml` / -f : filename)
  - Pod가 정상 실행 되는지 확인한다 (`kubectl get pods -w` / -w : watch (실시간으로 상태 변할때마다 자동으로 출력))

- ImagePullBackOff 에러는 오타 외에도 Private Registry 인증 정보(imagePullSecrets)가 누락했을 때에도 빈번히 발생한다.
- Pod가 정상적으로 뜨지 않을 때에는 **kubectl describe pod 명령어의 맨 아래 Events 섹션을 확인하자.**
