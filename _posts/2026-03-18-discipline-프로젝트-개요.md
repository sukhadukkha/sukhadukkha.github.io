## OLAP (Online Analytical Processing, 온라인 분석 처리)

- OLAP은 대규모 데이터를 다양한 각도에서 '빠르게 분석'하기 위한 기술 또는 시스템을 의미합니다.

- 핵심 특징: 단순히 데이터를 넣고 빼는 것보다, 수억 건 이상의 방대한 데이터에서 합계, 평균, 그룹화 등의 복잡한 계산을 순식간에 처리하는 것이 목표입니다. 주로 '컬럼 지향(Columnar)' 저장 방식을 사용하여 필요한 데이터만 골라 읽기 때문에 분석 속도가 매우 빠릅니다.

- 프로젝트에서의 역할: 대규모 로그, 트레이스(Trace), LLM 추론 데이터처럼 양이 많고 상세한 데이터를 장기 보관하고 분석하는 데 사용됩니다. "지난 한 달간 특정 모델의 평균 비용은 얼마인가?" 같은 복잡한 질문에 답을 내는 엔진 역할을 합니다.

- 대표적인 예시: 프로젝트에서 사용하는 ClickHouse(클릭하우스)가 고성능 OLAP 엔진에 해당합니다.

------------------------------------------------------------------------------------------------

## ClickHouse (클릭하우스)

- ClickHouse는 대규모 데이터를 실시간으로 분석하기 위해 만들어진 `오픈소스 컬럼 지향(Column-oriented) 데이터베이스 관리 시스템(DBMS)`입니다.

- 왜 쓰나요? (OLAP) 
    - 일반적인 DB(MySQL 등)가 한 줄씩 데이터를 넣고 빼는 데 최적화되어 있다면, ClickHouse는 수십억 건의 데이터 중 특정 항목(예: '전체 사용자의 평균 토큰 사용량')만 빠르게 뽑아내어 분석하는 OLAP(Online Analytical Processing)에 특화되어 있습니다.

- 이 프로젝트에서의 역할: 실시간 메트릭은 Prometheus가 담당하지만, 용량이 크고 상세한 로그, 트레이스, LLM 추론 데이터를 오랫동안 보관하고 빠르게 쿼리하기 위해 ClickHouse 아키텍처를 사용합니다.

- 장점: 압축률이 매우 높아 저장 공간을 아낄 수 있고, 데이터 처리 속도가 일반적인 행 지향 DB보다 수백 배에서 수천 배까지 빠릅니다.

## Gantt Chart (간트 차트)

- 간트 차트는 원래 프로젝트 일정 관리를 위해 사용하는 막대그래프 형태의 도구입니다. 하지만 IT 엔지니어링, 특히 분산 트레이싱(Distributed Tracing) 분야에서는 프로그램의 실행 흐름을 시각화하는 용도로 아주 중요하게 쓰입니다.

- 어떻게 생겼나요?: 세로축에는 실행된 작업(함수 호출, API 요청 등)이 나열되고, 가로축은 시간의 흐름을 나타냅니다. 각 작업은 가로 막대로 표시되어 언제 시작해서 언제 끝났는지 보여줍니다.

- 이 프로젝트에서의 역할: 복잡한 LLM 추론 과정과 애플리케이션 호출 과정을 스팬(Span, 작업 단위)으로 분해하여 간트 차트 형식으로 시각화합니다.

- 왜 유용한가요?: 여러 서비스가 얽혀 있을 때 어디서 시간이 가장 오래 걸렸는지(병목 지점) 한눈에 알 수 있습니다. 어떤 작업이 실패했는지, 어떤 순서로 호출되었는지 흐름을 파악하기 좋습니다.

- 요약하자면 ClickHouse는 "엄청나게 많은 데이터를 빠르게 분석하기 위한 저장소"이고, 간트 차트는 "그 데이터 속에 담긴 복잡한 실행 순서를 보여주는 지도"라고 이해하시면 됩니다.

## OTel(OpenTelemetry)이란?

- OTel은 시스템에서 발생하는 텔레메트리 데이터(Telemetry Data)인 메트릭, 로그, 트레이스를 수집, 변환, 전송하기 위한 오픈소스 프레임워크이자 표준 규격입니다. 과거에는 모니터링 도구마다 데이터 형식이 달라 도구를 바꿀 때마다 코드를 다 수정해야 했지만, OTel이 등장하면서 "어떤 도구를 쓰든 데이터는 이 표준 형식으로 보내자"라는 약속이 생겼습니다.

- 프로젝트에서의 구체적인 역할: 우리 프로젝트에서 OTel은 에이전트와 LLM 사이의 복잡한 통신을 데이터화하는 핵심 역할을 합니다.

- 데이터 표준화: 수집한 프롬프트, 응답, 토큰 사용량 등을 OTel GenAI Agent 시맨틱 컨벤션(표준 규격)으로 변환하여 전송합니다.

- 분산 트레이싱(Tracing): 에이전트가 LLM을 호출하고, 도구를 사용하고, 다시 응답을 받는 전체 과정을 스팬(Span) 단위로 쪼개서 기록합니다.

- 통합 관측: 이렇게 표준화된 데이터를 받아 ClickHouse 등에 저장하고 시각화합니다.

## 텔레메트리 아키텍처(Telemetry Architecture)란 

- 시스템의 상태를 파악하기 위해 발생하는 다양한 신호(Metrics, Logs, Traces)를 수집, 전송, 저장, 분석하는 전체적인 구조를 말합니다.

- 텔레메트리 아키텍처의 4단계 흐름
  - 데이터 생성 및 수집 (Collection): 에이전트나 애플리케이션에서 발생하는 LLM API 트래픽, 프롬프트, 토큰 사용량 등의 원천 데이터를 캡처합니다. 우리 프로젝트에서는 Proxy G/W가 이 역할을 수행하며, 별도의 코드 수정 없이 데이터를 뽑아냅니다.
  - 표준화 및 전송 (Processing & Export): 수집한 제각각의 데이터를 OpenInference나 OTel(OpenTelemetry) 같은 표준 규격으로 변환합니다. 이렇게 표준화해야 나중에 다양한 분석 도구에서 데이터를 공통적으로 사용할 수 있습니다.
  - 저장 (Storage): 데이터의 성격에 따라 분리하여 저장하는 하이브리드 저장소 전략을 취합니다. Prometheus: 실시간으로 변하는 수치 데이터(메트릭) 저장. ClickHouse: 용량이 큰 상세 기록(로그, 트레이스, 추론 전문)의 장기 보관 및 고속 조회.
  - 시각화 및 분석 (Visualization): 저장된 데이터를 꺼내어 관리자가 보기 편한 대시보드로 구성합니다. 단순 차트뿐만 아니라 서비스 간 호출 관계를 보여주는 서비스 그래프나 실행 순서를 보여주는 간트 차트 등으로 표현합니다.

## TSDB (Time Series Database, 시계열 데이터베이스)
   
- TSDB는 '시간의 흐름에 따라 발생하는 데이터(시계열 데이터)'를 저장하고 관리하는 데 최적화된 데이터베이스입니다.

- 핵심 특징: 데이터가 '언제(Timestamp)' 발생했는지가 가장 중요한 기준이며, 짧은 간격으로 계속해서 쌓이는 데이터를 빠르게 저장하고 조회하는 데 특화되어 있습니다.

- 프로젝트에서의 역할: 실시간 메트릭(Metric) 정보를 저장하는 데 사용됩니다. 예를 들어, 서버의 CPU 사용량 변화나 현재 LLM API의 호출 횟수 같은 수치 데이터를 실시간으로 쌓아두고 모니터링할 때 쓰입니다.

- 대표적인 예시: 프로젝트 문서에 언급된 Prometheus(프로메테우스)가 전형적인 TSDB입니다.

## eBPF (extended Berkeley Packet Filter)

- eBPF는 리눅스 커널 내부에서 샌드박스 형태로 프로그램을 실행할 수 있게 해주는 혁신적인 기술입니다. 원래는 네트워크 패킷 필터링용이었으나, 현재는 시스템 관측, 보안, 네트워킹의 핵심 기술로 자리 잡았습니다.

- 왜 쓰나요? (커널 레벨 관측): 기존 방식은 애플리케이션에 일일이 코드를 심어야 했지만, eBPF는 커널 레벨에서 데이터를 가로챕니다. 따라서 소스 코드 수정 없이도 프로세스들이 어떤 네트워크 통신을 하는지 완벽하게 파악할 수 있습니다.

- 프로젝트에서의 역할: 에이전트 프로세스와 다른 서비스들 사이의 실제 통신 관계를 관측하여, 서비스 간 연결 상태를 역동적인 Force Graph(망 형태의 그래프)로 시각화하는 기반이 됩니다.

## CSAP (Cloud Service Assurance Program)

- CSAP는 한국인터넷진흥원(KISA)에서 주관하는 '클라우드 보안인증제' 입니다. 공공기관에 클라우드 서비스를 제공하려는 기업은 반드시 이 인증을 받아야 합니다.

- 핵심 목적: 공공기관의 데이터를 안전하게 보호하기 위해 클라우드 서비스의 보안성을 평가하고 인증하는 것입니다.

- 평가 항목: 물리적 보안, 데이터 보호, 접근 제어 등 까다로운 보안 요건을 충족해야 합니다. 특히 금융권이나 공공기관을 타겟으로 하는 엔터프라이즈 솔루션에는 필수적입니다.

------------------------------------------------------------------------------------------------

## RBAC (Role-Based Access Control)

- RBAC는 '역할 기반 접근 제어'로, 개별 사용자에게 권한을 주는 것이 아니라 '역할(Role)'에 권한을 부여하고 사용자를 그 역할에 할당하는 방식입니다.

- 왜 중요한가?: 대규모 시스템(Enterprise)에서는 수백 명의 권한을 일일이 관리할 수 없습니다. "운영자", "분석가", "뷰어" 등 역할을 정의하고 관리함으로써 보안 사고를 방지하고 관리를 효율화합니다.

- 프로젝트에서의 역할: CSAP 인증이나 금융권의 망 분리 환경(인터넷망과 업무망 분리)처럼 엄격한 보안 규제에 대응하기 위해 고도화된 권한 관리 시스템을 구축하는 것입니다.

------------------------------------------------------------------------------------------------

## 폐쇄망(Air-gap) 환경과 설치 패키지

- Air-gap: 보안상의 이유로 외부 인터넷과 완전히 단절된 네트워크 환경을 의미합니다. 국방, 금융, 국가 주요 시설 등이 이에 해당합니다.

- 설치 패키지 개발: 인터넷이 안 되는 환경(On-premise 포함)에서도 우리 솔루션이 설치되고 동작할 수 있도록 모든 의존성 파일을 포함한 패키지(예: Helm Chart, Docker Image 등)를 제작하는 작업입니다.%
