## 상황

사용자들이 간헐적인 타임아웃을 보고하고 있습니다. 패킷 손실이 로컬 네트워크에서 발생하는지, 업스트림 게이트웨이에서 발생하는지, 외부 인터넷에서 발생하는지 구간별로 진단하세요.

- `업스트림 게이트웨이`
  - 로컬 게이트웨이가 외부 인터넷으로 나가기 위해 바로 다음으로 만나는 장비
  - 주로 ISP(인터넷 서비스 제공자, 예: KT, SKT, AWS IGW 등)의 라우터가 여기에 해당한다.
  - 로컬 게이트웨이: 공유기나 사무실의 L3 스위치 (내부망의 출구)
- 제공 데이터

- `default gateway`
  - 어디로 가야 할지 모르는 패킷을 보내는 기본 출구
- ping 192.168.1.1 -> 같은 네트워크 -> 직접 통신 (로컬)

- `ISP (Internet Service Provider)`: 인터넷을 윌에게 연결해주는 통신사(KT, SK ...)

## 해결

```
1. 기본 게이트웨이 확인

ip route | grep default

2. 구간별 ping 테스트 및 결과 저장

{
  echo "=== Gateway ==="
  ping -c 5 192.168.1.1 | tail -2
  echo ""
  echo "=== DNS (8.8.8.8) ==="
  ping -c 5 8.8.8.8 | tail -2
  echo ""
  echo "=== Internet (google.com) ==="
  ping -c 5 google.com | tail -2
} > /tmp/network_diagnostics.txt

3. 상세 경로 분석

trace route google.com

4. MTR로 지속적 모니터링 (패킷 손실 구간 정확히 파악)

mtr -rw -c 20 google.com

```


- `traceroute`: 패킷이 목적지까지 가는 경로를 단계별로 보여주는 도구
  - 패킷은 여러 라우터를 거쳐간다. `traceroute`는 이걸 하나씩 찍어낸다.
  - 목적지까지 가는 경로를 단계별로 보여준다.


- `MTR(My Trace Route)`
  - traceroute + ping 합친 상위 도구
  - 계속 반복해서 통계 보여줌
  - Loss% / AVG / Best,Worst 보여줌
  - `-r (report mode)`: 한 번 측정 후 결과만 출력 / 없으면 계속 갱신되는 화면
  - `-w (wide report)`: 출력 폭 넓게 (잘림 방지)
  - `-c 20 (count)`: 각 hop 마다 20번 측정 -> ping을 20번씩 보내서 통계 계산
  - `-i (interval)`: ping 보내는 간격 설정 


| 항목    | traceroute | mtr      |
| ----- | ---------- | -------- |
| 방식    | 1회 경로 확인   | 지속적 측정   |
| 패킷 손실 | 보기 어려움     | 정확히 표시   |
| 실무 활용 | 기본 확인      | 문제 분석 핵심 |

- `궁금한점`
  - 실습 화면에서의 Loss가 100%인데, 문제 생긴걸까?
- 해결
  - 결론은 문제가 아니다.
  - 그 라우터가 응답을 안해주는 것
  - 이유는? 라우터들은 종종 보안, 부하, 정책 이슈로 ICMP (ping/mtr) 응답을 차단하기 때문이다.
  - 마지막 목적지는 Loss가 0%로 정상이다.
