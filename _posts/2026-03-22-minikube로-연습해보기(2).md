## Ingress 실습해보기

![image1](/assets/images/Ingress.png)

- Step1: `Ingress Controller 활성화`
  - minikube는 Ingress Controller가 기본 비활성화기 때문에 켜줘야한다.

```
minikube addons enable ingress
-> 내부적으로 Nginx Ingress Controller 설치 (ingress-nginx namespace도 생성)

# 활성화 확인 (Pod가 Running 될 때까지 30초~1분 대기)
kubectl get pods -n ingress-nginx -w

# 모든 네임 스페이스 한번에 조회
kubectl get pods -A 
```

- Step2: 테스트용 앱 2개 배포 (apps.yml)
  - 2개의 deployment app-api, app-web
  - 2개의 Service app-api, app-web 생성

- deployment 적용 및 pod, svc 생성 확인

- `Ingress 리소스 작성(ingress.yml)`

```
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /   # 경로 prefix 제거 후 전달(myapp.local/api -> myapp.local/
spec:
  ingressClassName: nginx                           # minikube ingress addon 지정
  rules:
  - host: myapp.local                               # 가상 도메인 (hosts 파일에 등록할 것)
    http:
      paths:
      - path: /api
        pathType: Prefix                            # 앞에 /api 붙으면 다 보내라
        backend:
          service:
            name: svc-api
            port:
              number: 80
      - path: /web
        pathType: Prefix
        backend:
          service:
            name: svc-web
            port:
              number: 80
```

- ingress.yml 파일 적용
- 상태 확인 (`kubectl get ingress my-ingress`)

- Step3: hosts 파일 등록 & 테스트 (`Mac + Docker 환경에서는 192.168.49.2 직접 접근 안되서 의미 없음`)
  - minikube ip 통해 ip 확인
  - `sudo vi /etc/hosts` -> 192.168.49.2 myapp.local 추가

```
# /api 경로 → svc-api → App A
curl http://myapp.local/api
# 출력: Hello from API service

# /web 경로 → svc-web → App B
curl http://myapp.local/web
# 출력: Hello from WEB service
```

- 문제 발생
  - curl 명령 결과 안나오고 무한 로딩
- 해결 
  - `minikube tunnel` (minikube는 Ingress IP를 실제로 라우팅하려면 터널이 필요하다.) 해결 X
  - Ingress IP, hosts 파일, Ingress Controller Pod 상태 다 정상
  - Mac M 시리즈 특유 문제 -> `minikube가 Docker driver로 뜨면 192.168.49.2가 Mac 호스트에서 직접 라우팅이 안된다.`
  - `minikube service URL로 접근`
    - 새 터미널에서 `kubectl port-forward -n ingress-nginx service/ingress-nginx-controller 8080:80`
    - 또 다른 터미널에서 `curl -H "Host: myapp.local" http://localhost:8080/api`
  
```
curl -H "Host: myapp.local" http://localhost:8080/api
         ↑                          ↑
   헤더에만 존재               실제 접속 주소
   (Ingress 라우팅용)          (port-forward 터널)
```

```
내 터미널                    클러스터 내부
curl localhost:8080   →   ingress-nginx-controller:80   →   Ingress 라우팅   →   svc-api or svc-web
```

```

                                       5678 전달)
```

- Step 5: 심화 - 호스트 기반 라우팅
  - 경로 분기가 아닌 도메인 자체로 분기하는 패턴도 많이 사용한다. 
  - rules 추가 (ingress.yml)

```yaml

  - host: api.myapp.local       # 도메인으로 분기
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: svc-api
            port:
              number: 80
  - host: web.myapp.local
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: svc-web
            port:
              number: 80
```

### Ingress와 NodePort의 차이점

NodePort는 서비스마다 별도 포트가 필요하지만, Ingress는 L7 레이어에서 경로나 도메인으로 분기해서 여러 서비스를 포트 80/443 하나로 노출할 수 있다.
