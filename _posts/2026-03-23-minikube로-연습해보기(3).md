## RBAC(Role-Based Access Control)란? 

![img.png](/assets/images/RBAC.png)

- 역할(Role) 기반으로 권한을 관리하는 방식
- 구성요소
  - `Role (역할)`: 어떤 행동을 할 수 있는지 정의
  - `Subject (주체)`: 권한을 받는 대상 (User, Group, ServiceAccount)
  - `RoleBinding`: 누가 어떤 Role을 쓰는지 연결
- `RBAC 없으면?`
  - 누구나 서버 삭제 가능, 보안 붕괴
- `RBAC 있으면?`
  - 개발자 -> 읽기만
  - 운영자 -> 배포 가능
  - 관리자 -> 전체 권한

## RBAC 실습 해보기

- 모니터링 전용 SA 만들고, Pod/로그는 읽을 수 있지만, 삭제는 못하게 제한해보기

- `Step 1: Namespace 생성`
  - `kubectl create namespace monitoring`
- `Step 2: ServiceAccount(SA) 생성 및 적용`
  - `kubectl get serviceaccount -n monitoring`

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: monitoring-sa
  namespace: monitoring
```

- `Step 3: Role 생성 및 적용`
  - `verbs`: 권한
  - `kubectl get role -n monitoring`

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: monitoring
rules:
- apiGroups: [""]               # "" = core API group (Pod, Service 등)
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]  # 읽기만 가능
```

- 자주 쓰는 verb

| Verb   | 의미     |
| ------ | ------ |
| get    | 단건 조회  |
| list   | 목록 조회  |
| watch  | 실시간 감시 |
| create | 생성     |
| delete | 삭제     |
| update | 수정     |


- `Step 4: RoleBinding 생성 및 적용`
  - `kubectl get rolebinding -n monitoring`

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: monitoring-binding
  namespace: monitoring
subjects:                        # 누구에게 부여할지
- kind: ServiceAccount
  name: monitoring-sa
  namespace: monitoring
roleRef:                         # 어떤 Role을 부여할지
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

- `Step 5: 권한 검증`
  
```
# monitoring-sa는 Pod 조회 가능?
kubectl auth can-i get pods \
  --as=system:serviceaccount:monitoring:monitoring-sa \
  -n monitoring
# yes

# Pod 삭제는?
kubectl auth can-i delete pods \
  --as=system:serviceaccount:monitoring:monitoring-sa \
  -n monitoring
# no

# 다른 네임스페이스(default)에서는?
kubectl auth can-i get pods \
  --as=system:serviceaccount:monitoring:monitoring-sa \
  -n default
# no  ← Role은 네임스페이스 안에서만 유효
```

- `monitoring:monitoring-sa` 의미: 누가 요청했는지 / monitoring 네임스페이스에 있는 monitoring-sa 계정이 요청한다.
- `-n monitoring` 의미: 어디에 대해 요청하는지 / monitoring 네임스페이스의 pod를 조회하겠다.

- `Step 6: 실제 Pod에 SA 붙여보기`
  - Pod 내부에서 API 서버 직접 호출 (`kubectl exec -it monitor-pod -n monitoring -- sh`)
  - Pod 안에서 실행 (`kubectl get pods -n monitoring` / `kubectl delete pod monitor-pod -n monitoring`)
  - get pods는 성공, delete pod는 실패 확인 가능

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: monitor-pod
  namespace: monitoring
spec:
  serviceAccountName: monitoring-sa    # 여기서 SA 지정
  containers:
  - name: kubectl-container
    image: bitnami/kubectl
    command: ["sleep", "9999"]
```

- `Step 7: ClusterRole`
  - Role은 네임스페이스 안에서만 유요한데, 클러스터 전체 리소스(Node 등)를 봐야 하면 ClusterRole을 사용한다.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-reader
rules:
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: node-reader-binding
subjects:
- kind: ServiceAccount
  name: monitoring-sa
  namespace: monitoring
roleRef:
  kind: ClusterRole
  name: node-reader
  apiGroup: rbac.authorization.k8s.io
```

- kubectl auth can-i get nodes --as=system:serviceaccount:monitoring:monitoring-sa

## 흐름 정리

Pod 실행

-> monitoring-sa 로 API 서버에 요청<br>
-> RoleBinding이 SA(monitoring-sa)에 Role(pod-reader) 연결<br>
-> Role의 verbs 범위 안에서만 허용


- Pod가 API 서버에 요청할 때 SA 토큰을 쿠버네티스가 자동으로 Pod 내부에 마운트해준다.
  - ls /var/run/secrets/kubernetes.io/serviceaccount/
  - token -> API 서버 인증에 쓰는 JWT 토큰
  - ca.crt -> API 서버 인증서
  - namespace -> 현재 네임스페이스
- 예전 방식: SA 만들면 Secret에 토큰 자동 생성됨 -> SA 마다 토큰 Secret 존재했었음
- 현재 방식: `Pod가 생성 될때 SA 토큰 발급` -> 파일로 마운트
