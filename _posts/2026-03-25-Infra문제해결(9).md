## 상황

피크 시간대에 사용자 대면 서비스의 응답 속도가 급격히 느려졌습니다. 서버를 확인한 결과 devops 사용자가 실행한 백그라운드 데이터 처리 작업이 CPU를 85% 이상 점유하고 있습니다.<br>
이 작업을 중단할 수는 없지만 사용자 서비스에 영향을 줄여야 합니다.


- 제공 데이터

```
top 출력

top - 14:35:22 up 5 days, 3:20, 2 users, load average: 4.82, 4.15, 3.67

  PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM   TIME+ COMMAND
 5432 devops    20   0 2048576 256840  12340 R  85.3  12.4   8:23 data-processor
 1234 www-data  20   0  892340 125600  45670 S  12.1   6.1  45:12 nginx
 2345 postgres  20   0 1245680 345200  67890 S   8.5  16.8  32:45 postgres
------------------------------------------------------------------------------------------------

ps -o pid,user,ni,comm -p 5432

  PID USER      NI COMMAND
 5432 devops     0 data-processor
```

- `ps -o(output Fields) pid,user,ni,comm`
- `ni(Nice Value)`: 스케줄링 우선순위 (-20~19) / 낮을수록 우선 할당
- `comm(command Name)`: 실행중인 명령어의 이름만 간단히 보여준다.
- ps -e(Every)f(Full-formant)


## 해결

- 사용자 대면 서비스가 필요한 CPU 자원을 확보받지 못하고 있다.
- data-processor 프로세스의 nice값이 0으로, nginx와 postgres 같은 서비스와 동일한 우선순위


```
1. 현재 프로세스 우선순위 확인

ps -o pid,user,ni,comm -p 5432

2. nice 값 10으로 변경 (우선순위 낮춤)

renice 10 -p 5432

3. 변경 확인 

ps -o pid,user,ni,comm -p 5432
```

- 백그라운드 배치 작업은 처음부터 `nice -n 10 ./data-processor`로 낮은 우선순위 부여하고 실행하자.
- crontab에서도 `nice -n 10 /usr/local/bin/batch-job.sh`처럼 사용할 수 있다. 더 강력한 리소스 제어가 필요하면 cgroups를 활용하여 CPU 사용률 상한을 직접 설정할 수 있다.
  - `crontab`: 리눅스에서 특정 시간에 특정 작업을 자동으로 실행해 주는 스케줄러
    - -e(edit): 크론 설정 편집
    - -l(list): 크론 목록 확인
    - -r(remove): 현재 사용자의 모든 크론 설정 삭제
  - `cgroups(Control Groups)`: 리눅스 커널이 제공하는 기능으로, 프로세스들의 자원 사용량(CPU,메모리,네트워크 등)을 강제로 제한할 수 있다. 
    - 한계치 설정
  
- crontab 작성 규칙

```
# 분  시  일  월  요일  [실행할 명령어]
* * * * * /home/user/script.sh
```

- 예시

```
매일 새벽 3시에 배치 작업 실행 (부하 방지를 위해 nice 사용)
00 03 * * * nice -n 10 /usr/local/bin/batch-job.sh

매주 월요일 새벽 5시에 로그 파일 삭제
00 05 * * 1 rm /var/log/myapp/*.log

평일(월~금) 오전 9시에만 알림 서비스 실행
00 09 * * 1-5 /home/devops/send_alert.sh
```
