## 상황

운영 환경 배포 후 애플리케이션에 심각한 버그가 발견되어 이전 버전으로 롤백해야 합니다. 그러나 Docker 이미지 레지스트리를 확인한 결과 모든 이미지가 동일한 태그로 되어 있어 어떤 커밋 버전의 이미지인지 추적할 수 없습니다. 빌드 워크플로우를 분석하여 원인을 찾으세요.

- 제공 데이터

```yaml
# ./github/workflows/build.yml
name: Build and Push

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t myapp:latest .

      - name: Push Docker image
        run: docker push myapp:latest
```

```
docker images 출력

REPOSITORY    TAG       IMAGE ID       CREATED          SIZE
myapp         latest    a1b2c3d4e5f6   2 minutes ago    245MB

배포 히스토리

2024-03-01 09:00  배포 v1 (정상) - myapp:latest
2024-03-02 10:00  배포 v2 (정상) - myapp:latest
2024-03-03 11:00  배포 v3 (장애) - myapp:latest   ← 현재
# v2로 롤백하고 싶지만 해당 이미지를 식별할 수 없음
```


## 해결

- 문제
  - workflows에서 Docker 이미지를 항상 latest 태그로만 빌드하고 있다.
  - latest는 가장 최근 빌드를 가리키는 가변 태그이므로, 새 이미지가 push 되면 이전 이미지에 대한 참조가 사라진다.
  - 이로 인해 특정 커밋에 해당하는 이미지 식별 불가, 롤백 시 이전 버전의 이미지 찾기 불가, 운영 환경의 이미지가 어떤 버전인지 추적 불가 문제 발생

- 해결 방법(build.yml 파일 수정)

```yaml
name: Build and Push

on:
  push:
    branches: [main]    # main branch에 push 될 때 워크플로우 실행

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set short SHA
        run: echo "SHORT_SHA=${GITHUB_SHA::7}" >> $GITHUB_ENV  # GITHUB_SHA : github이 자동으로 제공하는 현재 커밋의 전체 SHA(40자)

      - name: Build Docker image
        run: |
          docker build \
            -t myapp:${{ env.SHORT_SHA }} \
            -t myapp:latest \
            .

      - name: Push Docker image         # abc1234 태그로 이미지 push, latest 태그로 이미지 push
        run: |
          docker push myapp:${{ env.SHORT_SHA }}      
          docker push myapp:latest                    

```

```
# 롤백 시 특정 커밋의 이미지로 즉시 복구 가능
kubectl set image deployment/myapp myapp=myapp:abc1234
```

- `${GITHUB_SHA::7}`: GITHUB_SHA에서 앞 7자리만 잘라냄 -> ex) abc1234
- `>> $GITHUB_ENV`: 잘라낸 값을 환경변수로 등록
- 이미지 태그 두 개 붙여서 push (abc1234 -> 커밋 추적용, latest -> 최신버전)

- 실무 팁: Docker 이미지 태깅에는 Git commit SHA를 기본으로 사용하고, latest는 보조 태그로 함께 붙이기.
  - 이미지 레이블에 전체 SHA를 기록하면 감사 추적에도 도움이 된다.

- 빌드 시 이미지 레이블에 SHA 주입하는 법
  
```bash
# SHA 먼저 뽑고
export FULL_SHA=$(git rev-parse HEAD)
export SHORT_SHA=$(git rev-parse --short HEAD)

# 태그에도 동일하게 사용
docker build \
  --label org.opencontainers.image.revision=$FULL_SHA \
  -t myapp:$SHORT_SHA \
  .
```

- Github Actions 흐름

```yaml
- name: Set short SHA
  run: echo "SHORT_SHA=${GITHUB_SHA::7}" >> $GITHUB_ENV

- name: Build Docker image
  run: |
    docker build \
      --label org.opencontainers.image.revision=${{ github.sha }} \
      -t myapp:${{ env.SHORT_SHA }} \
      -t myapp:latest \
      .
```

- 롤백 시나리오

- 현재 배포된 이미지 확인

```
kubectl get deployment myapp -o wide

myapp   3/3     3            3           myapp        myapp:f3a9b2c

#                            -o [형식] 
wide: 기본 출력 + 추가 컬럼
yaml: YAML 형식으로 출력
json: json 형식으로 출력
name: 리소스 이름만 출력
```

- 롤백 필요 -> 이전 커밋 SHA 확인

```
git log --oneline

a1b2c3d 장바구니 기능 추가 (롤백 목표)

```


- 해당 커밋 이미지로 롤백


## 배운 점 

- Docker 이미지 태그를 latest 단독으로 관리하면 롤백 시 어느 버전으로 돌아가야 할지 알 수 없음.
- Git commit SHA를 태그로 붙이면 버전 추적과 롤백이 가능하다.
- `$GITHUB_SHA`는 Github Actions runner가 자동으로 주입하는 환경 변수라 별도 등록 없이 바로 사용 가능하다.
- 롤백 흐름을 익혔다.
  - `git log --oneline` -> 롤백할 커밋 SHA 확인
  - `kubectl set image` -> 해당 SHA 이미지로 변경
  - `kubectl rollout status` -> 롤백 진행 상태 확인
  - `kubectl get deployment -o wide` -> 이미지 태그 변경 최종 확인

- 롤백 명령어들 

```bash
# 1. 롤백할 커밋 SHA 확인
git log --oneline

# 2. 해당 커밋 이미지로 롤백
kubectl set image deployment/myapp myapp=myapp:abc1234

# 3. 롤백 상태 확인
kubectl rollout status deployment/myapp

# 4. 빠르게 이전 버전으로 되돌리기 (SHA 모를 때)
kubectl rollout undo deployment/myapp

# 5. 특정 revision으로 되돌리기
kubectl rollout history deployment/myapp        # revision 목록 확인
kubectl rollout undo deployment/myapp --to-revision=2
```
