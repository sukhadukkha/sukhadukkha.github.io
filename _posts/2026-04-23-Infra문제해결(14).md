## 해결

- 원인
  - logs의 Error -> `CACHE_TTL` 환경 변수 설정 누락
  - `kubectl describe pod` 출력의 환경변수 목록에는 CACHE_TTL 존재 X

```
1. ConfigMap에 cache_ttl 키가 있는지 확인해보기

kubectl get configmap api-config -o yaml

2. 키가 없으면 ConfigMap에 추가

kubectl patch configmap api-config --type merge -p '{"data":{"cache_ttl":"300"}}' / or edit configmap으로 파일에 직접 추가

3. Deployment에 환경변수 매핑 추가

kubectl edit deployment api-server
# env 섹션에 추가:
#   - name: CACHE_TTL
#     valueFrom:
#       configMapKeyRef:
#         name: api-config
#         key: cache_ttl

4. 롤아웃 재시작
kubectl rollout restart deployment api-server

5. 정상 동작 확인
kubectl rollout status deployment api-server
kubectl get pods -l app=api-server
```


## 배운점

- - CrashLoopBackOff 트러블슈팅 순서를 익혔다.
  - `kubectl get pods` -> CrashLoopBackOff 상태 확인
  - `kubectl describe pod` -> 환경변수 설정 확인
  - `kubectl logs` -> 실제 에러 메시지 확인
  - 로그에서 원인 특정해보기
- 새 버전 배포 시 코드에서 필요한 환경변수가 추가되었다면 ConfigMap/Secret도 반드시 함께 업데이트해야 한다.
- ConfigMap을 수정해도 이미 떠있는 Pod는 자동으로 재시작되지 않는다. `kubectl rollout restart`로 Pod를 재생성해야 새 값을 읽어온다.
- 애플리케이션 코드에서 선택적 환경변수에 기본값을 설정하면 ConfigMap 누락 시에도 에러 없이 동작할 수 있다. `const cacheTTL = process.env.CACHE_TTL || "300"`
- Helm이나 Kustomize를 사용하면 Deployment와 ConfigMap을 하나의 릴리스로 묶어서 배포할 수 있어 누락을 방지할 수 있다.
  - 그럼 여기서 `Helm은 무엇인지 + 사용 예시를 알아보자`

---

- `Helm`: Kubernetes용 패키지 매니저 (apt, npm과 같은 개념) / `helm install nginx` (Kubernetes에 nginx 설치)
- 애플리케이션을 Helm chart로 패키징해서 관리한다.
- 왜 필요한가? 
  - 따로따로 관리하고 적용하던 파일들을 하나로 묶어서 관리

```bash
kubectl apply -f deployment.yaml
kubectl apply -f configmap.yaml
kubectl apply -f service.yaml
# 파일 하나라도 빠뜨리면 문제 발생
```

- 구조

```
mychart/
├── Chart.yaml           # 차트 메타정보
├── values.yaml          # 변수 정의 (환경별로 바꿔쓰는 값)
└── templates/
    ├── deployment.yaml  # 템플릿
    ├── configmap.yaml
    └── service.yaml
```

- yaml 파일들 예시

```yaml
# Chart.yaml
apiVersion: v2
name: myapp          # 차트 이름
description: "My API Server"
version: 1.0.0       # 차트 자체 버전
appVersion: v2.1.0   # 실제 앱 버전

# values.yaml

# 여기서 정의한 값들이 templates/ 안에 주입됨
image:
  repository: registry.example.com/api-server
  tag: v2.1.0

replicas: 2

config:
  cache_ttl: "300"
  redis_url: "redis-master.internal:6379"

resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"

# templates/ - 실제 Kubernetes 리소스 정의 , values.yaml의 값을 {{ .Values.xxx }}로 사용
# templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-api     # Release 이름 자동 주입
spec:
  replicas: {{ .Values.replicas }}   # values.yaml에서 가져옴
  template:
    spec:
      containers:
        - name: api
          image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
          resources:
            requests:
              cpu: {{ .Values.resources.requests.cpu }}
              memory: {{ .Values.resources.requests.memory }}

# templates/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ .Release.Name }}-config
data:
  cache_ttl: {{ .Values.config.cache_ttl }}
  redis_url: {{ .Values.config.redis_url }}

# templates/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: {{ .Release.Name }}-svc
spec:
  selector:
    app: {{ .Release.Name }}
  ports:
    - port: 80
      targetPort: 8080
```

- 여기서 `{{ .Release.xxx }}`는 Helm이 자동으로 생성하는 정보 (`helm install` 할 때 자동으로 생성함)
- 개발 / prod 환경 별 값들 다르게 적용 예시

```bash
# 개발환경
helm install myapp ./mychart -f values-dev.yaml

# 운영환경
helm install myapp ./mychart -f values-prod.yaml
```

- 실제 배포 흐름

```bash
# 1. 차트 구조 생성 (mychart 폴더 자동 생성)
helm create mychart
```

```bash
# 2. 배포 전 렌더링 확인 (실제 배포 X, 미리보기)
helm template mychart ./mychart
```

```bash
# 3. 실제 배포
helm install myapp ./mychart

NAME: myapp
LAST DEPLOYED: Mon Jan 15 10:00:00 2024
STATUS: deployed
```

```bash
# 4. values.yaml 수정 후 업그레이드
# cache_ttl: "300" → "600" 으로 변경
helm upgrade myapp ./mychart

Release "myapp" has been upgraded. Happy Helming!
STATUS: deployed
REVISION: 2          # ← 버전 자동 증가
```

```bash
# 4. 배포된 리소스 확인
helm get manifest myapp
```

```yaml
# 실제 렌더링된 결과 출력
apiVersion: v1
kind: ConfigMap
data:
  cache_ttl: "600"    # 반영됨
---
apiVersion: apps/v1
kind: Deployment
...
```

```bash
# 5. 문제 생기면 롤백
helm rollback myapp 1   # revision 1로 롤백

Rollback was a success! Happy Helming!
```
