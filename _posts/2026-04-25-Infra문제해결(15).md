## 상황

서버가 느려지고 사용자들이 타임아웃 에러를 경험하고 있습니다. nginx 사용자로 여러 워커 프로세스가 실행 중인데, 그 중 하나가 비정상적으로 높은 CPU를 소비하고 있는 것으로 의심됩니다. 문제의 프로세스를 식별하세요.

- 제공 데이터

```
ps aux --sort=%cpu | head -10

USER       PID   %CPU  %MEM    VSZ    RSS TTY  STAT START   TIME COMMAND
nginx     12345  92.3   4.1 245678  42340 ?    R    08:15  45:12 nginx: worker process
python3   22310  45.7   6.2 567890 123450 ?    S    07:00  32:45 python3 app.py
java      19872  37.9  12.4 1234560 256780 ?   Sl   06:30  28:30 java -jar app.jar
nginx     12346  15.4   2.8 245678  28560 ?    S    08:15   8:23 nginx: worker process
nginx     12347   1.2   1.0 245678  10240 ?    S    08:15   0:45 nginx: worker process
nginx     12340   0.0   0.5 245678   5120 ?    Ss   08:15   0:02 nginx: master process
root         1   0.0   0.1  16924   3240 ?    Ss   Mar01   0:05 /sbin/init

nginx 설정(발췌)

worker_processes auto;
worker_connections 1024;
```

## 해결

- nginx 사용자의 프로세스 중 12345 프로세스의 CPU가 92.3%로 굉장히 높은 CPU 사용률을 가지고있다.

```
1. 문제 프로세스 PID 식별

ps -u nginx -o pid,%cpu,%mem,comm --sort=-%cpu | head -5
- nginx user의 프로세스 pid, cpu, mem, comm 만 모아서 출력
-o : 출력할 정보 고르는 옵션
comm: 실행 명령어 이름

2. 해당 프로세스 상세 정보 확인

ls -l /proc/12345/fd | wc -l -> wc -l (wc: Word Count, -l : Lines / 줄 수 출력)
cat /proc/12345/status -> 프로세스 상태 확인

Name:   nginx
Pid:    12345
Threads: 1
VmRSS:  42340 kB      # 실제 메모리 사용량
VmPeak: 98765 kB      # 최대 메모리 사용량
voluntary_ctxt_switches: 100       # 자발적 컨텍스트 스위칭
nonvoluntary_ctxt_switches: 98000  # 비자발적 (높으면 CPU 경합 심함)

3. 문제 지속 시 해당 워커 종료 (master가 자동 재생성)
kill -QUIT 12345  -> -QUIT (종료 요청 + 로그 요청)
```

- 실무 팁
  - Nginx 워커 프로세스는 master 프로세스가 관리한다. 문제 워커를 kill 하면 master가 새 워커를 자동 생성한다.
  - 특정 워커의 CPU가 비정상적으로 높으면 strace -p <PID>로 시스템 콜을 추적하여 원인 분석할 수 있다.

```bash
strace -p 12345           # 실행 중인 프로세스 추적
strace -p 12345 -c        # 시스템 콜 통계 요약
strace -p 12345 -e read   # read 콜만 필터링
```

```
strace -p 출력 예시

epoll_wait(5, [], 512, 0)  = 0
read(4, "GET /video/big.mp4 HTTP/1.1\r\n", 8192) = 8192
read(4, "...", 8192) = 8192
read(4, "...", 8192) = 8192   # 계속 read 반복 → 대용량 파일 처리 중
write(3, "...", 8192) = 8192
...

-c 옵션 출력 

% time     seconds  calls  syscall
 78.32      45.123  98432  read      # read가 78% → 파일 읽기가 병목
 15.21       8.765   1234  write
  6.47       3.234    432  epoll_wait
```



## 배운점

- Nginx 워커 프로세스 중 하나만 CPU가 비정상적으로 높을 때
  전체 nginx를 재시작할 필요 없이 해당 워커 PID만 kill하면 된다.
  master 프로세스가 새 워커를 자동으로 재생성하기 때문이다.

- `/proc/[PID]` 디렉토리로 실행 중인 프로세스의 상세 정보를 확인할 수 있다.
    - `fd/` -> 열린 파일/소켓 목록 (파일 디스크립터 누수 확인)
    - `status` -> 메모리 사용량, 컨텍스트 스위칭 횟수 확인

- `strace -p [PID]`로 프로세스가 커널에 어떤 시스템 콜을 하는지
  실시간으로 추적할 수 있다.
    - `read`가 반복되면 대용량 파일 처리가 원인
    - 특정 콜이 무한 반복되면 무한루프 의심
    - `-c` 옵션으로 시스템 콜 통계를 요약해서 병목 지점을 빠르게 특정할 수 있다.

- 트러블슈팅은 넓은 범위에서 좁혀가는 순서로 진행한다.
    - `ps aux` -> 문제 프로세스 특정
    - `/proc/[PID]` -> 프로세스 상태 확인
    - `strace` -> 시스템 콜 레벨에서 근본 원인 분석
