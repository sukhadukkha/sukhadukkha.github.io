# [nginx] nginx + Docker 리버스 프록시 실습

## 실습 환경

- OS: Ubuntu 22.04 aarch64 (Oracle Cloud A1.Flex)
- nginx: 호스트에 직접 설치
- docker: 컨테이너로 앱 서빙

---

## 목표 구조

```
외부 요청 → 호스트 nginx (80) → docker 컨테이너 (8080)
외부에서 8080 직접 접근 ❌
```

---

## 1. nginx 설정 파일 구조

```
/etc/nginx/
├── nginx.conf              # 메인 설정 파일
├── sites-available/        # 가상 호스트 설정 보관함 (비활성)
├── sites-enabled/          # 실제 적용되는 설정 (심볼릭 링크)
├── conf.d/                 # 추가 설정 파일
├── mime.types              # 파일 확장자별 MIME 타입
└── snippets/               # 재사용 가능한 설정 조각
```

**sites-available vs sites-enabled**
- `sites-available` - 설정 파일 보관함. 여기 있다고 바로 적용되지 않음
- `sites-enabled` - 실제 적용되는 설정. sites-available에서 심볼릭 링크로 연결

---

## 2. nginx.conf 핵심 설정

```nginx
user www-data;              # nginx 실행 유저 (보안상 root 아닌 www-data 사용)
worker_processes auto;      # CPU 코어 수에 맞게 워커 프로세스 자동 설정

events {
    worker_connections 768; # 워커 프로세스 하나당 최대 동시 연결 수
}

http {
    sendfile on;            # 파일 전송 최적화
    gzip on;                # 응답 데이터 압축

    access_log /var/log/nginx/access.log;   # 요청 로그
    error_log /var/log/nginx/error.log;     # 에러 로그

    include /etc/nginx/sites-enabled/*;     # 가상 호스트 설정 불러오기
}
```

## 4. nginx 로그 확인

```bash
# 실시간 접근 로그 확인
sudo tail -f /var/log/nginx/access.log

# 실시간 에러 로그 확인
sudo tail -f /var/log/nginx/error.log
```

**로그 레벨**

| 레벨 | 의미 |
|------|------|
| `[notice]` | 일반 동작 알림 (정상) |
| `[warn]` | 경고 |
| `[error]` | 에러 |
| `[crit]` | 심각한 에러 |

**access.log 형식**
```
IP주소 - - [날짜] "메서드 경로 프로토콜" 상태코드 바이트 "레퍼러" "User-Agent"
```

---

## 5. nginx 프로세스 관리

```bash
sudo systemctl start nginx    # 시작
sudo systemctl stop nginx     # 중지
sudo systemctl restart nginx  # 재시작 (기존 연결 끊김)
sudo systemctl reload nginx   # 설정만 다시 읽음 (기존 연결 유지) ← 권장
sudo systemctl status nginx   # 상태 확인
```

**설정 파일 문법 검사 (적용 전 필수)**
```bash
sudo nginx -t
```

> 운영 중인 서버에서 설정 변경 시 restart 대신 reload 사용. restart는 잠깐 서비스가 끊김.

## 6. Docker 컨테이너 실행

```bash
docker run -d -p 8080:80 --name test-nginx nginx
```

| 옵션 | 설명 |
|------|------|
| `-d` | 백그라운드 실행 |
| `-p 8080:80` | 호스트 8080 → 컨테이너 내부 80 포트 매핑 |
| `--name` | 컨테이너 이름 지정 |

**포트 충돌이 없는 이유**

컨테이너는 독립된 네트워크 네임스페이스를 가짐. 호스트 80포트와 컨테이너 내부 80포트는 완전히 별개.

```
호스트 80  → 호스트 nginx
호스트 8080 → 컨테이너 내부 80 (docker nginx)
```

## 7. 리버스 프록시 설정

`/etc/nginx/sites-available/default` 수정

```nginx
location / {
    proxy_pass http://localhost:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}
```

| 설정 | 역할 |
|------|------|
| `proxy_pass` | 요청을 8080으로 전달 |
| `proxy_set_header Host` | 원래 호스트 정보 전달 |
| `proxy_set_header X-Real-IP` | 클라이언트 실제 IP 전달 |

> `X-Real-IP`가 필요한 이유: 리버스 프록시를 거치면 앱 서버 입장에서 클라이언트 IP가 localhost로 보임. 원래 IP를 헤더에 담아서 전달.

설정 적용
```bash
sudo nginx -t
sudo systemctl reload nginx
```

## 8. 8080 포트 외부 차단

리버스 프록시 설정 후 외부에서 8080 직접 접근을 막아야 함.

**ufw에서 차단**
```bash
sudo ufw delete allow 8080
```

**Oracle NSG에서도 8080 인바운드 규칙 삭제**

결과
```
외부 → 80 (nginx) → 8080 (컨테이너) ✅
외부 → 8080 직접 접근 ❌
```

## 리버스 프록시를 사용하는 이유

1. **포트 통일** - 사용자에게 80, 443만 노출. 내부 포트 숨김
2. **여러 서비스 라우팅** - 하나의 서버에서 여러 앱 운영 가능
```
/api  → 8080 (백엔드)
/     → 3000 (프론트엔드)
```
3. **SSL 한 곳에서 관리** - nginx에서만 SSL 처리
4. **로드밸런싱** - 여러 컨테이너에 트래픽 분산

## 트러블슈팅

### 한글 깨짐
**원인**: HTML 파일에 charset 설정 없음
**해결**: `<meta charset="UTF-8">` 추가

### 브라우저 접속 안 됨
**원인 1**: Oracle NSG에서 포트 미개방
**원인 2**: ufw에서 포트 차단
**원인 3**: iptables REJECT 규칙 존재

iptables 확인
```bash
sudo iptables -L INPUT -n -v
```

REJECT 규칙 삭제
```bash
sudo iptables -D INPUT -j REJECT --reject-with icmp-host-prohibited
```
