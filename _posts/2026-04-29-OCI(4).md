## 웹 애플리케이션 개발 README

# Sweet Story

둘만의 추억을 날짜, 사진, 장소와 함께 기록하는 프라이빗 메모리 웹 앱입니다. React 프론트엔드와 Spring Boot 백엔드를 분리해 개발했고, Docker 기반으로 OCI Compute Instance에 배포했습니다. 사진 파일은 서버 디스크가 아니라 OCI Object Storage에 저장합니다.

이 프로젝트는 단순 CRUD 앱을 넘어, 개인 서비스 하나를 직접 설계하고 배포하면서 인증, 파일 저장, 컨테이너 배포, CI/CD, 클라우드 권한 설정까지 경험하는 것을 목표로 했습니다.

## 주요 기능

- 로그인 기반의 비공개 추억 공간
- JWT Access Token / Refresh Token 인증
- D-day 시작일 설정 및 홈 화면 카운터
- 추억 작성, 상세 조회, 전체 타임라인
- 날짜, 날씨, 장소 정보 기록
- 추억 카테고리 생성, 수정, 삭제 및 필터링
- 사진 업로드, 갤러리 조회, 사진 카테고리 관리
- Google Maps 기반 장소 검색 및 지도에서 추억 확인
- OCI Object Storage 업로드 및 PAR URL 기반 이미지 조회

## 현재 배포 구조

현재 운영 구조는 OCI Compute Instance 한 대에서 Docker Compose로 프론트엔드, 백엔드, MySQL 컨테이너를 실행하는 방식입니다. 사진 파일은 OCI Object Storage에 저장합니다.

```text
Browser
  |
  | HTTP :80
  v
OCI Compute Instance
  |
  v
memory-nginx container
  |  React 정적 파일 서빙
  |  /api/* reverse proxy
  v
memory-backend container
  |  Spring Boot REST API
  +-- memory-mysql container
  |
  +-- OCI Object Storage
```

컨테이너 역할:

| Container | Role |
|---|---|
| `memory-nginx` | React 빌드 결과물 서빙, `/api/*` 요청을 백엔드로 프록시 |
| `memory-backend` | Spring Boot REST API |
| `memory-mysql` | 애플리케이션 데이터 저장 |

현재 구조는 `Compute + Docker Compose + Object Storage` 기반입니다. 추후 관리형 DB를 붙이면 더 명확한 3-Tier 구조로 확장할 수 있습니다.

## CI/CD

GitHub Actions로 `main` 브랜치 push 시 CI 검증과 배포를 자동화했습니다.

```text
main push
  -> Frontend CI
  -> Backend CI
  -> Docker image build/push
  -> OCI 서버에서 docker compose pull
  -> docker compose up -d
```

워크플로우 파일:

```text
.github/workflows/deploy.yml
```

CI 단계:

- Frontend: `npm ci`, `npm run lint`, `npm run build`
- Backend: MySQL service container 실행 후 `./gradlew test`

배포 단계:

- backend Docker image build/push
- frontend nginx Docker image build/push
- OCI 서버에서 최신 이미지 pull
- Docker Compose로 컨테이너 재기동

## Object Storage 사용 방식

사진 업로드 시 백엔드가 OCI Object Storage에 파일을 저장하고, 브라우저가 접근할 수 있는 PAR(Pre-Authenticated Request) URL을 생성합니다.

```text
React
  -> multipart/form-data 업로드
  -> Spring Boot Photo API
  -> OCI Object Storage PutObject
  -> PAR URL 생성
  -> DB에 photo metadata 저장
  -> 브라우저가 PAR URL로 이미지 직접 로드
```

배포 환경에서는 OCI Instance Principal을 사용해 Compute Instance 자체에 Object Storage 접근 권한을 부여합니다. 별도 OCI API Key를 컨테이너에 넣지 않는 방향으로 구성했습니다.

# CI/CD 트러블슈팅

## [CI/CD] GitHub Actions ARM64 빌드 시간 단축 트러블슈팅

## 환경
- OCI A1.Flex (ARM64 / aarch64)
- GitHub Actions
- Docker

---

## 문제 상황

매 빌드마다 15분 소요. 캐시 설정(`cache-from: type=gha`)이 있음에도 개선되지 않음.

```yaml
- name: Build and push backend
  uses: docker/build-push-action@v6
  with:
    platforms: linux/arm64
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

---

## 원인

GitHub Actions 기본 runner는 **x86_64** 아키텍처야.

OCI 서버가 **ARM64(aarch64)** 인데 `platforms: linux/arm64`로 설정하면 GitHub runner에서 **QEMU 에뮬레이션**으로 ARM64 이미지를 빌드해야 해서 엄청 느려짐.

```
GitHub runner (x86_64) → QEMU 에뮬레이션 → ARM64 이미지 빌드 → 15분
```

캐시가 있어도 크로스 플랫폼 빌드 자체가 느려서 의미없음.

---

## 해결 방법 - Self-hosted Runner 설정

OCI 서버(ARM64)에 GitHub Actions runner를 직접 설치해서 네이티브로 빌드.

```
OCI 서버 (ARM64) → 네이티브 ARM64 빌드 → 빠름
```

### 1. OCI 서버에 runner 설치

GitHub 레포 → Settings → Actions → Runners → New self-hosted runner → **Linux / ARM64** 선택

```bash
mkdir ~/actions-runner && cd ~/actions-runner

# ARM64용 runner 다운로드
curl -o actions-runner-linux-arm64-2.334.0.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.334.0/actions-runner-linux-arm64-2.334.0.tar.gz

tar xzf ./actions-runner-linux-arm64-2.334.0.tar.gz

# GitHub 레포와 연결
./config.sh --url https://github.com/{username}/{repo} --token {TOKEN}
```

### 2. 서비스로 등록 (재부팅해도 자동 시작)

```bash
sudo ./svc.sh install
sudo ./svc.sh start
sudo ./svc.sh status
```

### 3. yml 파일 수정

deploy job의 `runs-on`을 `self-hosted`로 변경.
SSH 배포 단계 제거 (runner가 OCI 서버에 있으니까 바로 실행 가능).

```yaml
deploy:
  name: Build Images and Deploy
  runs-on: self-hosted  # ubuntu-latest → self-hosted
  needs: [frontend, backend]
  if: github.event_name == 'push' && github.ref == 'refs/heads/main'

  steps:
    - name: Build and push backend
      uses: docker/build-push-action@v6
      with:
        platforms: linux/arm64  # 이제 네이티브 빌드
        push: true
        ...

    - name: Deploy  # SSH 없이 바로 실행
      run: |
        cd ~/app
        docker compose pull
        docker compose up -d
        docker image prune -f
```

---

## Self-hosted Runner 동작 원리

GitHub → OCI 방향으로 명령이 가는 게 아니라 **OCI 서버가 GitHub을 폴링**하는 방식.

```
GitHub에 push 발생
→ GitHub이 "잡 있어요" 신호 전송
→ OCI 서버 runner가 신호 감지 (Listening for Jobs)
→ OCI 서버에서 네이티브로 잡 실행
```

---

## Self-hosted Runner 주의사항

| 항목 | 내용 |
|------|------|
| 보안 | Private 레포에서만 사용 권장 |
| 디스크 관리 | `docker image prune -f` 주기적으로 실행 |
| 서버 재부팅 | `svc.sh install`로 서비스 등록하면 자동 시작 |
| 동시성 | 서버 1대 = 동시 빌드 1개 |

---

## 결과

| 방식 | 빌드 시간 |
|------|-----------|
| GitHub hosted runner (x86_64 + QEMU) | 15분 |
| Self-hosted runner (ARM64 네이티브) | 단축 예상 |

- **실제 빌드 3분 33초로 약 12분 단축 성공!**

# 배포 트러블슈팅 기록

## 1. nginx 413 Request Entity Too Large (사진 업로드)

**증상**: 사진 업로드 시 413 에러

**원인**: nginx 기본 요청 크기 제한 1MB

**해결**: `frontend/nginx.conf`에 `client_max_body_size` 추가

```nginx
server {
    listen 80;
    client_max_body_size 50M;
    ...
}
```

---

## 2. Docker Compose 설치 (Ubuntu ARM64)

**증상**: `docker: unknown command: docker compose`

**원인**: Ubuntu apt 패키지의 Docker에는 compose 플러그인이 포함되지 않음

**해결**: GitHub에서 aarch64 바이너리 직접 설치

```bash
mkdir -p ~/.docker/cli-plugins
curl -SL https://github.com/docker/compose/releases/latest/download/docker-compose-linux-aarch64 \
  -o ~/.docker/cli-plugins/docker-compose
chmod +x ~/.docker/cli-plugins/docker-compose
```

---

## 3. OCI Object Storage 403/404 (사진 업로드)

**증상**: `POST /api/memories/{id}/photos` 요청 시 403 반환

**백엔드 에러**:
```
BmcException: BucketNotFound - Either the bucket named 'my-memory-bucket' does not exist
in the namespace 'idvp2o3599xs' or you are not authorized to access it
```

**원인**: OCI Instance Principal 미설정 — 컴퓨트 인스턴스에 Object Storage 접근 권한 없음

> OCI는 권한 없을 때 "버킷 없거나 권한 없음"으로 404와 403을 같이 반환함

**해결 순서**:

### 3-1. 컴퓨트 인스턴스 OCID 확인
서버에서:
```bash
curl -s http://169.254.169.254/opc/v2/instance/ -H "Authorization: Bearer Oracle" | python3 -m json.tool | grep '"id"'
```

### 3-2. Dynamic Group 생성
OCI Console → Identity & Security → 도메인 → Default → 동적 그룹 → 생성

- **이름**: `memory-app-instances`
- **일치 규칙** (따옴표 필수):
```
instance.id = 'ocid1.instance.oc1.iad.xxxxx'
```

### 3-3. Policy 생성
OCI Console → Identity & Security → 정책 → 생성

- **이름**: `memory-app-storage-policy`
- **Compartment**: root (tenancy)
- **정책 문** (수동 편집기 사용):
```
Allow dynamic-group 'Default'/'memory-app-instances' to manage object-family in tenancy
```
> AWS에서는 EC2에 S3 권한 IAM Role을 붙이면 되는데, OCI에서는 먼저 Compute 인스턴스를 Dynamic Group에 넣고, 그
> Dynamic Group에 Object Storage Policy를 부여해야 합니다.
> `object-family` = objects + buckets + preauthenticated-requests 전부 포함  
> `manage objects in tenancy`만 쓰면 PAR 생성 권한이 빠져서 추가로 실패함

**주의사항**:
- Dynamic Group 규칙에 따옴표 없으면 인스턴스 매칭 안 됨
- `manage preauthenticated-requests` 는 OCI 정책에서 유효하지 않은 리소스 타입 → `object-family` 사용
- Policy 적용에 수 분 소요될 수 있음

## 4. Spring Boot 시작 전 nginx 요청 전달 (502)

**증상**: 서비스 시작 직후 일시적으로 502 Bad Gateway

**원인**: `depends_on`이 컨테이너 시작만 기다리고 Spring Boot가 완전히 뜨기 전에 nginx가 요청 전달

**해결** (선택): `docker-compose.yml`에 backend healthcheck 추가

```yaml
backend:
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
    interval: 10s
    timeout: 5s
    retries: 5

nginx:
  depends_on:
    backend:
      condition: service_healthy
```
