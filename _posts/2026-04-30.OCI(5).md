# [SSL] 와일드카드 SSL 인증서 발급 (*.monithub.org)

## 환경
- OS: Ubuntu 22.04 aarch64 (Oracle Cloud A1.Flex)
- 도메인: *.monithub.org (와일드카드)
- 도구: Certbot + Let's Encrypt

---

## 배경

### SSL 인증서가 필요한 이유
- HTTPS 통신을 가능하게 해주는 파일
- "이 서버가 진짜 monithub.org 서버가 맞다"는 신분증 역할
- Docker Registry는 기본적으로 HTTPS만 신뢰

### 와일드카드 인증서란
일반 인증서는 특정 도메인 하나만 커버하지만 와일드카드 인증서는 모든 서브도메인을 커버함.

```
*.monithub.org 인증서
├── registry.monithub.org ✅
├── blog.monithub.org ✅
└── anything.monithub.org ✅
```

이 상태에서 네 서버의 Nginx 가 “어떤 서브도메인인지” 보고 컨테이너로 나눠줍니다.

예:

```
registry.monithub.org → Docker Registry
api.monithub.org → Spring API
jenkins.monithub.org → Jenkins
```

### DNS 구성
```
*.monithub.org → OCI 퍼블릭 IP (와일드카드 A 레코드)
```

---

## 인증서 발급 과정

### 1. certbot 설치
```bash
sudo apt install certbot -y
```

### 2. 와일드카드 인증서 발급 명령어
```bash
sudo certbot certonly \
  --manual \
  --preferred-challenges dns \
  -d "*.monithub.org"
```

| 옵션 | 설명 |
|------|------|
| `certonly` | 인증서만 발급 (nginx 설정 건드리지 않음) |
| `--manual` | 수동으로 진행 |
| `--preferred-challenges dns` | DNS TXT 레코드로 도메인 소유권 증명 |
| `-d "*.monithub.org"` | 와일드카드 인증서 발급 |
| `-d "monithub.org"` | 루트 도메인도 포함 |

### 3. DNS TXT 레코드 추가

certbot 실행 시 아래와 같이 TXT 레코드 추가 요구.

```
타입: TXT
이름: _acme-challenge.monithub.org
값1: {첫 번째 값}
```

> 와일드카드 인증서는 DNS 방식으로만 발급 가능.
> 도메인 소유자만 DNS 설정 가능하므로 도메인 관리자에게 TXT 레코드 추가 요청 필요.

### 4. DNS 전파 확인 후 엔터
```bash
dig TXT _acme-challenge.monithub.org
```

값이 보이면 엔터 눌러서 진행.


## 주의사항

**자동 갱신 안 됨**

`--manual` 방식은 자동 갱신이 지원되지 않음. 만료 전에 수동으로 재발급 필요.

```bash
sudo certbot certonly \
  --manual \
  --preferred-challenges dns \
  -d "*.monithub.org" \
  -d "monithub.org"
```

만료일 2주 전에 재발급 권장.

## 문제 1. ERR_CERT_COMMON_NAME_INVALID

### 증상
```
net::ERR_CERT_COMMON_NAME_INVALID
```

### 원인
내가 발급한 와일드카드 인증서(`*.monithub.org`)가 아닌 팀장님 서버의 인증서(`monithub.org`만 커버)가 서빙되고 있었음.

### 확인 방법
```bash
# 실제 서빙되는 인증서 확인
curl -v --insecure https://registry.monithub.org 2>&1 | grep "subject\|expire"

# 인증서 SAN 확인
sudo openssl x509 -in /etc/letsencrypt/live/monithub.org/fullchain.pem -noout -text | grep -A 2 "Subject Alternative"
```

### 해결
팀장님에게 와일드카드 인증서(`*.monithub.org`) 발급 요청.

---

## 문제 2. 팀장님 인증서 교체 후에도 SAN 없음

### 증상
팀장님이 인증서 tar 파일을 줬는데 압축 해제 후 확인하면 여전히 `monithub.org`만 있음.

### 원인
첫 번째 tar 파일(`monithub-cert.tar.gz`)은 와일드카드가 아닌 기존 인증서였음.

두 번째 tar 파일(`monithub.org-wildcard-letsencrypt-20260429-170931.tgz`)이 와일드카드 인증서.

### 확인 방법
```bash
# tar 파일 내용 확인
sudo tar tzf /tmp/monithub.org-wildcard-letsencrypt-20260429-170931.tgz
```

### 해결
두 번째 tar 파일 압축 해제 후 nginx conf에서 인증서 경로 변경.

```bash
# 로컬에서 서버로 파일 전송
scp -i ~/.ssh/ssh-key-2026-04-22.key \
  /Users/jihopark/Downloads/monithub.org-wildcard-letsencrypt-20260429-170931.tgz \
  ubuntu@129.159.177.124:/tmp/

# 서버에서 압축 해제
cd /
sudo tar xzf /tmp/monithub.org-wildcard-letsencrypt-20260429-170931.tgz
```

**인증서 경로가 기존과 다름 주의**
```
기존: /etc/letsencrypt/live/monithub.org/
새로운: /etc/letsencrypt/live/monithub.org-wildcard/
```

nginx conf 수정.
```nginx
ssl_certificate /etc/letsencrypt/live/monithub.org-wildcard/fullchain.pem;
ssl_certificate_key /etc/letsencrypt/live/monithub.org-wildcard/privkey.pem;
```

## 문제 3. nginx 시작 실패 - host not found in upstream

### 증상
```
host not found in upstream "registry" in /etc/nginx/conf.d/default.conf:32
```

### 원인
`proxy_pass http://registry:5000`으로 설정했는데 `registry` 이름의 Docker 컨테이너가 없어서 nginx가 시작조차 못함.

### 해결
Docker Registry 컨테이너 띄우기 전까지는 `localhost:5000`으로 설정.

```nginx
proxy_pass http://localhost:5000;
```

> Docker Registry 컨테이너 띄운 후 컨테이너 이름으로 변경 필요.

## 문제 4. registry.monithub.org 접속 시 Monithub 랜딩페이지 뜸

## 최종 nginx 설정

```nginx
server {
    listen 80;
    server_name _;

    client_max_body_size 50M;

    root /usr/share/nginx/html;
    index index.html;

    location /api/ {
        proxy_pass http://backend:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_read_timeout 60s;
    }

    location / {
        try_files $uri $uri/ /index.html;

server {
    listen 443 ssl;
    server_name registry.monithub.org;

    ssl_certificate /etc/letsencrypt/live/monithub.org-wildcard/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/monithub.org-wildcard/privkey.pem;

    location / {
        proxy_pass http://localhost:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

# [Docker Registry] Private Registry 구축 작업 정리

## 환경
- OS: Ubuntu 22.04 aarch64 (Oracle Cloud A1.Flex)
- 도메인: registry.monithub.org
- nginx: Docker 컨테이너로 운영

---

## 목표 구조

```
사용자 → https://registry.monithub.org
         ↓ nginx (SSL 처리)
         
         /v2/ 요청 → monithub-registry:5000 (Registry API)
         / 요청    → monithub-registry-web:80 (GUI)
```

---

## 작업 내용

### 1. htpasswd 인증 정보 생성

```bash
sudo apt install apache2-utils -y
sudo mkdir -p /data/docker/auth
htpasswd -Bbn monithub123 '!monithub123()' | sudo tee /data/docker/auth/htpasswd
```

### 2. docker-compose.yml에 registry 서비스 추가

```yaml
  registry:
    image: registry:2
    container_name: monithub-registry
    volumes:
      - /data:/var/lib/registry
      - /data/docker/auth:/auth
    environment:
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: "Monithub Registry"
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd
      REGISTRY_HTTP_HEADERS_Access-Control-Allow-Origin: '[https://registry.monithub.org]'
      REGISTRY_HTTP_HEADERS_Access-Control-Allow-Methods: '[HEAD,GET,OPTIONS,DELETE]'
      REGISTRY_HTTP_HEADERS_Access-Control-Allow-Credentials: '[true]'
      REGISTRY_HTTP_HEADERS_Access-Control-Allow-Headers: '[Authorization,Accept,Cache-Control]'
      REGISTRY_HTTP_HEADERS_Access-Control-Expose-Headers: '[Docker-Content-Digest]'
      REGISTRY_HTTP_HEADERS_Access-Control-Max-Age: '[1728000]'
      REGISTRY_HTTP_HEADERS_X-Content-Type-Options: '[nosniff]'
    restart: unless-stopped

  registry-web:
    image: joxit/docker-registry-ui:latest
    container_name: monithub-registry-web
    environment:
      - REGISTRY_URL=https://registry.monithub.org
      - SINGLE_REGISTRY=true
    depends_on:
      - registry
    restart: unless-stopped
```

### 3. nginx conf에 registry 서버 블록 추가

```nginx
server {
    listen 443 ssl;
    server_name registry.monithub.org;

    ssl_certificate /etc/letsencrypt/live/monithub.org-wildcard/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/monithub.org-wildcard/privkey.pem;

    location /v2/ {
        proxy_pass http://monithub-registry:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location / {
        proxy_pass http://monithub-registry-web:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

**증상**

# [Docker Registry] docker push 405 에러 트러블슈팅

## 환경
- OS: Ubuntu 22.04 aarch64 (Oracle Cloud A1.Flex)
- nginx: Docker 컨테이너로 운영
- Registry: registry.monithub.org (HTTPS)

---

## 문제 상황

`docker push registry.monithub.org/test:latest` 실행 시 아래 오류 발생.

```
unknown: failed commit on ref "layer-sha256:...": 
unexpected status from PUT request to 
http://registry.monithub.org/v2/test/blobs/uploads/...: 405 Not Allowed
```

---

## 원인

에러 메시지에 `http://registry.monithub.org`로 PUT 요청이 가고 있는 걸 확인.

HTTPS로 접속했는데 registry가 HTTP로 리다이렉트해서 405 오류 발생.

nginx에서 registry 컨테이너로 프록시할 때 `X-Forwarded-Proto` 헤더가 없어서 registry가 클라이언트가 HTTPS로 접속했는지 몰랐던 것.

---

## 해결 방법

nginx conf `/v2/` location 블록에 헤더 및 옵션 추가.

```nginx
location /v2/ {
    proxy_pass http://monithub-registry:5000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;  # 추가 - HTTPS 정보 전달
    proxy_read_timeout 900;                       # 추가 - 큰 이미지 push 시 timeout 방지
    client_max_body_size 0;                       # 추가 - 이미지 크기 제한 없음
}
```

nginx 재시작.

```bash
docker compose restart nginx
```

---

## 각 설정 설명

| 설정 | 설명 |
|------|------|
| `proxy_set_header X-Forwarded-Proto $scheme` | 클라이언트가 HTTPS로 접속했다는 정보를 registry에 전달. 없으면 registry가 HTTP로 응답해서 405 발생 |
| `proxy_read_timeout 900` | nginx가 registry 응답 기다리는 시간. 큰 이미지 push 시 timeout 방지 |
| `client_max_body_size 0` | 업로드 크기 제한 없음. 기본값 있으면 큰 이미지 push 시 차단될 수 있음 |

---

## 결과

```bash
docker push registry.monithub.org/test:latest
# 성공 ✅

docker pull registry.monithub.org/test:latest
# 성공 ✅
```

## nginx 트래픽 흐름

### nginx conf 구조
```nginx
server {
    listen 443 ssl;
    server_name registry.monithub.org;

    location /v2/ {
        proxy_pass http://monithub-registry:5000;  # Registry API
    }

    location / {
        proxy_pass http://monithub-registry-web:80;  # GUI
    }
}
```

nginx는 URL 경로를 보고 어디로 보낼지 결정해. `/v2/`가 더 구체적인 경로라 우선순위가 높아.

### 브라우저 접속 시 (`https://registry.monithub.org`)

```
사용자 브라우저
    ↓ https://registry.monithub.org/
    ↓ nginx location / 매칭
    → http://monithub-registry-web:80 (GUI 화면 반환)
    
GUI에서 이미지 목록 조회 시
    ↓ https://registry.monithub.org/v2/_catalog
    ↓ nginx location /v2/ 매칭
    → http://monithub-registry:5000 (이미지 목록 반환)
```

### docker push 시

```
docker push registry.monithub.org/test:latest
    ↓ HTTPS 443 포트로 연결
    ↓ PUT /v2/test/blobs/uploads/... 요청
    ↓ nginx location /v2/ 매칭
    → http://monithub-registry:5000
    → /data/docker/registry에 이미지 레이어 저장
```

### docker pull 시

```
docker pull registry.monithub.org/test:latest
    ↓ HTTPS 443 포트로 연결
    ↓ GET /v2/test/manifests/latest 요청
    ↓ nginx location /v2/ 매칭
    → http://monithub-registry:5000
    → /data/docker/registry에서 이미지 레이어 반환
```

### location 우선순위

nginx는 가장 구체적인 경로를 먼저 매칭해.

```
/v2/_catalog  → location /v2/ 매칭 ✅ (더 구체적)
/             → location / 매칭 ✅
/api/test     → location / 매칭 ✅ (/v2/로 시작 안하니까)
```

## Docker Registry API 표준 경로

`/v2/`는 Docker가 표준으로 정한 Registry API 경로야. GUI 이미지(`joxit/docker-registry-ui`)가 이 표준에 맞춰서 API 호출함.

```
GET /v2/                      → API 버전 확인
GET /v2/_catalog              → 전체 이미지 목록
GET /v2/{이미지}/tags/list    → 특정 이미지 태그 목록
GET /v2/{이미지}/manifests    → 이미지 메타데이터
GET /v2/{이미지}/blobs        → 이미지 레이어
```

브라우저로 `https://registry.monithub.org` 접속하면 GUI가 로드되고, GUI가 바로 `/v2/_catalog`로 이미지 목록 요청을 날림. 개발자 도구 Network 탭에서 확인 가능.

## GUI에서 이미지 삭제 기능 활성화

기본적으로 삭제 기능이 비활성화되어 있음. 아래 설정 추가 필요.

### registry-web 환경변수 추가

```yaml
  registry-web:
    image: joxit/docker-registry-ui:latest
    container_name: monithub-registry-web
    environment:
      - REGISTRY_URL=https://registry.monithub.org
      - SINGLE_REGISTRY=true
      - DELETE_IMAGES=true  # 삭제 버튼 활성화
```

### registry 환경변수 추가

```yaml
  registry:
    environment:
      REGISTRY_STORAGE_DELETE_ENABLED: "true"  # 삭제 허용
```

적용 후 `docker compose up -d` 하면 GUI에서 삭제 버튼 활성화됨.
