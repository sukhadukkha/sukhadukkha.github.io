## 상황 

서버에서 간헐적인 지연과 타임아웃이 발생하고 있습니다. CPU와 메모리는 정상 수준이지만 디스크 I/O 사용률이 비정상적으로 높습니다. 사용자 파일 접근이 극도로 느려졌습니다. I/O 병목을 유발하는 프로세스를 찾아 조치하세요.

- 제공 데이터

```
iotop 

Total DISK READ:      125.43 M/s | Total DISK WRITE:      89.32 M/s

  TID  PRIO  USER      DISK READ  DISK WRITE  SWAPIN     IO>    COMMAND
 5678  be/4  backup     98.45 M/s   67.23 M/s  0.00 %  95.32 % rsync /data /backup
 3421  be/4  postgres   15.23 M/s   12.45 M/s  0.00 %  18.45 % postgres: vacuum
 8234  be/4  appuser     8.45 M/s    3.21 M/s  0.00 %  12.10 % /usr/bin/log-processor
 

iostat -x 1 1

Device   r/s     w/s   rkB/s    wkB/s  %util
sda    2450.0  1890.0 128440.0  91320.0  99.8

```

- `iotop` 출력 결과의 각 열 의미
  - Total DISK READ / WRITE: 시스템 전체에서 발생하는 실제 디스크 읽기/쓰기 속도의 총합입니다. 현재 초당 약 125MB 읽고, 89MB 쓰고 있습니다.
  - TID (Thread ID): 디스크 I/O를 일으키고 있는 개별 스레드의 식별 번호입니다. (프로세스 ID인 PID와 유사하지만 스레드 단위입니다.)
  - PRIO (Priority): I/O 스케줄링 우선순위입니다. be/4는 Best-effort 클래스의 우선순위 4(기본값)를 뜻합니다.
  - DISK READ / WRITE: 해당 스레드가 초당 읽고(READ) 쓰고(WRITE) 있는 데이터의 양입니다.
  - SWAPIN: 프로세스가 부족한 메모리를 보충하기 위해 스왑(SWAP) 메모리로부터 데이터를 읽어오는 데 쓰는 시간의 비율입니다. 현재는 모두 0.00%로 메모리 부족으로 인한 스왑 병목은 아닙니다.
  - IO>: 해당 스레드가 I/O 작업(읽기/쓰기 완료)을 기다리며 대기하는 데 소비한 시간의 백분율입니다. rsync가 95.32%라는 것은 디스크가 너무 바빠서 처리 요청이 계속 밀리고 있다는 뜻입니다.
  - COMMAND: 디스크 I/O를 유발한 실제 명령어(프로세스 이름)입니다.
- `iostat -x 1 1` 명령어 및 결과 의미
  - iostat은 디스크 장치 자체의 성능 통계를 보여준다.
  - `-x` : 기본 정보 외에 성능 분석에 필수적인 확장 통계 지표 함께 출력
  - `1 1` : 첫번째 1은 통계를 1초 간격으로 갱신, 두 번째 1은 그 1초 간격의 출력을 딱 1번만 하고 명령어를 종료

| 항목 | 의미 | 현재 수치 및 해석 |
|---|---|---|
| Device | 스토리지 장치 이름 | `sda` (첫 번째 SCSI/SATA 하드디스크 또는 EBS 볼륨 등) |
| r/s | 초당 디스크 읽기 요청 횟수 (Read Requests Per Second) | `2450.0회` (초당 디스크에 2,450번의 읽기 명령이 들어감) |
| w/s | 초당 디스크 쓰기 요청 횟수 (Write Requests Per Second) | `1890.0회` (초당 디스크에 1,890번의 쓰기 명령이 들어감) |
| rkB/s | 초당 디스크에서 읽은 데이터 양 (KB 단위) | `128,440 KB/s` (약 125.4 MB/s, iotop 총합과 일치) |
| wkB/s | 초당 디스크에 쓴 데이터 양 (KB 단위) | `91,320 KB/s` (약 89.1 MB/s) |
| %util | 디스크가 I/O 요청을 처리하느라 분주했던 시간의 비율 (Device Utilization) | `99.8%` (디스크가 쉴 새 없이 100% 가동 중이며 완전히 포화 상태) |

## 해결

- 5678 스레드가 I/O의 대부분을 소비하고있다. 
  - postgres 같은 중요 서비스와 동일한 우선순위를 가지고있다. best-effort, 기본값 4 (be /4)
  - iostat에서 디스크 사용률이 99.8%로 디스크가 포화상태이다.
  - 백업 작업이 DB와 애플리케이션의 디스크 접근을 방해하고있다.

- 해결 방법

```
1. I/O 과다 사용 프로세스 확인

iotop or iotop -o -b -n 1

2. 비핵심 프로세스의 I/O 우선순위 변경(idle)

ionice -c 3 -p 5678

3. log processor도 변경

ionice -c 3 -p 8234

4. 변경 확인

iotop
```

- iotop 의 각 옵션들 설명
  - -o (--only) : 현재 I/O 발생시키는 프로세스만 출력
  - -b (--batch) : 텍스트 모드로 출력
  - -n 1 (--iter=1) : 1번만 화면에 출력

- 백업 작업은 처음부터 idel로 설정하자.
- `crontab에서도 CPU와 IO 우선순위를 동시에 낮출 수 있다. (ionice -c 3 nice -n 10 ...)`
- I/O 클래스는 1(Realtime), 2(Best-effort), 3(Idle) 순서로 구분된디.
