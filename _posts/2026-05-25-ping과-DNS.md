## ping 동작 과정 알아보기

- `ping 8.8.8.8` / `ping www.google.com`
    - 차이점
      - DNS Server / Web Server
      - DNS Server 세팅이 안되어있다면 www.google.com 은 동작하지 않는다.
      - 만약 IP주소로는 ping이 나가는데, 도메인 주소로는 안된다면? DNS 설정이 꼬인 상태인 것.
    - 공통점 
      - 둘다 외부망
- 도메인 주소로의 ping은 IP주소가 출력된다.
- `ping -> ICMP 패킷(테스트 전용)`
  - www.naver.com -> ICMP패킷 응답 안해줌, www.google.com -> 응답 해줌
- tshark를 통해 ip 추적 해보기
  - `sudo apt install tshark`
- 패킷은 네트워크 인터페이스를 통해 통신하기에`ip addr` 명령 통해 네트워크 인터페이스 확인
  - 인터페이스란? 리눅스에서는 네트워크 장치를 전부 인터페이스라고 부른다.
  - ens160 -> 리눅스가 인식한 가상 네트워크 카드(NIC, 랜카드) 이름.

```
2: ens160: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    link/ether 00:0c:29:db:86:6e brd ff:ff:ff:ff:ff:ff
    altname enp2s0
    inet 192.168.29.141/24 metric 100 brd 192.168.29.255 scope global dynamic ens160
       valid_lft 910sec preferred_lft 910sec
    inet6 fe80::20c:29ff:fedb:866e/64 scope link 
       valid_lft forever preferred_lft forever
```

- `sudo tshark -i ens160 -f "icmp" -Y "ip.dst == 142.251.153.119"`
  - ens160 인터페이스의 목적지가 142 ... 인 ICMP 패킷만 캡처 
  - -i ens160 : 어떤 인터페이스에서 패킷 캡쳐할지 지정
  - -f (Filter) : 캡처 단계에서 필터 적용, 커널 레벨(BPF)에서 동작
  - -Y "ip.dst == ..." : 캡처된 패킷 중 화면에 보여줄 것만 필터링

- `sudo tshark -i ens160 -f "icmp" -Y "ip.dst == ... or ip.src == ..."`

## DNS 동작과정 추적하기

- `nmcli` -> NetworkManager를 제어하는 CLI 명령어, 리눅스 네트워크 설정(IP, DNS, 인터페이스 등)을 관리하는 도구
- `nmcli dev show` -> device show -> 현재 네트워크 인터페이스들의 상세 설정 출력
  - VMware환경에서 ens160 unmanaged 상태 -> `cat /etc/resolv.conf로 DNS 확인`

![img.png](/assets/images/resolvconf.png)

- 왜 호스트를 뜻하는 127.x.x.x 일까? 
- 이 주소는 `systemd-resolved` 라는 로컬 DNS 프록시의 주소이다.
- 흐름은 다음과 같다.
  - 내 시스템이 먼저 /etc/resolv.conf 확인 후 127.0.0.53에게 질의
  - systemd-resolved라는 리눅스 내부 서비스가 요청을 받고, 캐싱하고있던 주소인지 확인
  - 캐시히트가 일어나지 않으면 이때 외부 DNS에 질의한다.
  - DNS 캐싱을 통해 효율적으로 요청을 처리한다.
  - `resolvectl status` 명령 통해 원본 DNS 주소 확인 가능. 

```
Link 2 (ens160)
    Current Scopes: DNS
         Protocols: +DefaultRoute -LLMNR -mDNS -DNSOverTLS DNSSEC=no/unsupported
Current DNS Server: 192.168.29.2
       DNS Servers: 192.168.29.2
        DNS Domain: localdomain
```

- 왜 192.168.29.2일까? 내 공유기의 내부 주소는 192.168.0.1이다.
  - 이것은 실제 공유기가 아니라 VMware가 가상으로 생성한 NAT DNS Proxy다.
  - ubuntu VM(192.168.29.141) -> DNS 요청 -> 192.168.29.2(NAT) -> Host OS의 DNS 설정 참조 -> ISP DNS(210.~~)
  - 응답은 210.~ -> host os -> vmware DNS Proxy에게 전달 -> ubuntu에게 전달

- ping 명령어가 실제 DNS 이용하는지 확인해보기
  - `sudo tshark -i ens160 -Y dns` / ping www.google.com

![img.png](/assets/images/tshark4.png)

- 결과 : 192.168.29.141(ubuntu) -> 192.168.29.2 (DNS Proxy, ipv4,ipv6 주소 요청) -> www.google.com IP 주소 찾음 -> 142.~~ 주소 응답까지 확인 완료

- `sudo tcpdump -xxi ens160 arp`

## ping www.google.com 하면 벌어지는 일

- gateway의 MAC주소 찾기 (ARP)
- DNS (UDP) / 도메인의 실제 IP주소 알아내기 위해 
- ICMP
