## 상황 

원격 서브넷 10.20.0.0/16에 접근해야 하지만, 현재 라우팅 테이블에 해당 경로가 없어 통신이 불가능합니다. 영구적인 설정 변경 없이 임시로 라우팅 경로를 추가하여 접근을 확보하세요.

- 제공 데이터

```
ip route 출력 

default via 192.168.1.1 dev eth0
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.50
172.16.0.0/16 via 192.168.1.254 dev eth0

ping 테스트 

$ ping -c 1 10.20.1.100
connect: Network is unreachable

ip link show

1: lo: <LOOPBACK,UP> mtu 65536
2: eth0: <BROADCAST,MULTICAST,UP> mtu 1500
3: veth0: <BROADCAST,MULTICAST,UP> mtu 1500
```

- default via 192.168.1.1 dev eth0
  - Default Gateway 설정 
  - 모르는 주소로 통신하려면 eth0 랜카드를 통해 192.168.1.1로 패킷을 던져라.

- 192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.50
  - 이 리눅스 서버의 주소는 192.168.1.50 이고, 192.168.1.X 대역을 쓰는 같은 컴퓨터들과는 별도의 라우터 없이 eth0으로 직접 통신 가능.

- 172.16.0.0/16 via 192.168.1.254 dev eth0
  - 만약 목적지가 172.16.X.X 대역이라면 디폴트 게이트웨이로 가지 말고, 192.168.1.254 라는 특수 라우터에게 패킷을 던져라.
- Network is unreachable -> 라우팅 테이블을 확인해봐도 10.20.X.X와 통신하려면 어느 인터페이스로 가야하는지 모르겠다는 응답.

- `ip link show`
  - 이 서버에 꽂혀있는 물리적/논리적 네트워크 카드(인터페이스) 목록


## 해결 

- 현재 라우팅 테이블에 10.20.0.0/16 서브넷에 대한 경로가 없다.

- 10.20.0.0/16 서브넷과 통신하려면 라우팅 경로를 추가해줘야한다.

```
# 1. 현재 라우팅 테이블에서 10.20.0.0 확인
ip route | grep 10.20
# (출력 없음 = 경로 없음)

# 2. 임시 정적 경로 추가
ip route add 10.20.0.0/16 via 192.168.1.1 dev veth0

# 3. 경로 추가 확인
ip route | grep 10.20
# 10.20.0.0/16 via 192.168.1.1 dev veth0

# 4. 연결 테스트
ping -c 3 10.20.1.100

# 경로 삭제 (작업 완료 후)
ip route del 10.20.0.0/16 via 192.168.1.1 dev veth0

# 참고: ip route add로 추가한 경로는 재부팅 시 자동 삭제됩니다.
# 영구 설정이 필요하면 /etc/netplan/ 또는 /etc/network/interfaces에 추가합니다.
```

- `ip route add 10.20.0.0/16 via 192.168.1.1 dev veth0`
  - 목적지 주소가 10.20.X.X로 시작한다면, 192.168.1.1 이라는 라우터에게 패킷을 보내라. 
  - 그 라우터에게 패킷을 보내기 위해 veth0 이라는 랜카드로 패킷을 내보내라.
  - 메모리에만 임시로 저장되므로, 재부팅하면 사라진다.

- veth0 -> 도커나 가상화 환경에서 쓰이는 인터페이스
  - 10.x.x.x 대역(사설 대역)은 주로 가상망, VPN 혹은 쿠버네티스 파드 네트워크로 쓰인다.
  - eth0 밖에는 10.20.x.x 대역이 존재하지 않으니, 가상 네트워크에 가상 라우터(192.168.1.1)이 대기하고 있을 것.
  - 그렇기에 veth0 사용

- 흐름 정리
  - default는 eth0 인터페이스 통해 192.168.1.1 라우터로 패킷 전달
  - 172.16.0.0 대역으로 갈 패킷은 eth0 인터페이스를 통해 192.168.1.254 라우터로 패킷 전달
  - 10.20.0.0 대역으로 갈 패킷은 가상 veth0 인터페이스를 통해 192.168.1.1 가상 라우터로 패킷 전달

- 라우팅 흐름 이해하기 쉽게 정리

```
[상황 1] 구글(8.8.8.8)로 패킷을 보낼 때
1. 커널: "8.8.8.8은 특별한 지름길이 없네? default 규칙을 따르자."
2. 커널: "default 규칙을 보니 'eth0' 문으로 나가서 '192.168.1.1'한테 주라고 되어 있군!"
3. 결과: 패킷이 eth0 문을 열고 나가 복도에 있는 1.1 라우터에게 배달됨 ➔ 외부 인터넷 성공!

[상황 2] 지호님이 사내망(172.16.1.5)으로 패킷을 보낼 때
1. 커널: "어, 172.16.X.X 대역이네? 전용 이정표가 있다!"
2. 커널: "이정표를 보니 똑같이 'eth0' 문으로 나가되, 이번에는 '192.168.1.254'한테 주라고 되어 있네?"
3. 결과: 패킷이 똑같은 eth0 문으로 나가지만, 복도 끝에 있는 1.254 라우터에게 배달됨 ➔ 사내망 통신 성공!

[상황 3] 지호님이 가상망(10.20.1.100)으로 패킷을 보낼 때
1. 커널: "10.20.X.X 대역이구나! 우리가 임시로 추가해 둔 규칙을 보자."
2. 커널: "이번에는 'veth0' 문(지하 통로)으로 나가서 거기 있는 '192.168.1.1'한테 주라고 되어 있네!"
3. 결과: 패킷이 eth0 문을 쳐다보지도 않고 veth0 지하 통로로 쏙 들어가서 가상 라우터에게 배달됨 ➔ 가상망 통신 성공!
```

- `물리 네트워크 영역`
  
```
default via 10.0.0.1 dev enp0s6 ... src 10.0.0.10 metric 100 
10.0.0.0/24 dev enp0s6 proto kernel scope link src 10.0.0.10 metric 100 
10.0.0.1 dev enp0s6 proto dhcp scope link src 10.0.0.10 metric 100
```

- 인터페이스 : enp0s6
- private IP: 10.0.0.10
- 게이트웨이: 10.0.0.1
- 일반적인 인터넷 및 물리 네트워크 통신은 enp0s6 인터페이스를 통해 10.0.0.1 로 나간다.

- `WireGuard 터널링 영역`

```
10.0.0.48 dev wg0 scope link 
10.96.0.0/12 dev wg0 scope link 
10.200.0.0/24 dev wg0 proto kernel scope link src 10.200.0.2
10.244.0.0/16 dev wg0 scope link
```

- 인터페이스 : wg0
- 내 VPN IP : 10.200.0.2
- wg0 라는 가상 보안 통로.
  - 10.200.0.0/24 : WireGuard로 묶인 노드들끼리 사설 통신을 하기 위한 VPN 네트워크 망, 내 주소는 .2
  - 10.96.0.0/12 : 쿠버네티스의 Service 가상 IP 대역, 쿠버네티스 서비스로 향하는 트래픽을 다른 노드로 안전하게 전달하기 위해 wg0 터널로 태우고 있다.
    - 쿠버네티스 Cluster IP 대역 글로벌 표준 기본 값이 `10.960.0./12` 이다.
  - 10.244.0.0/16 : 쿠버네티스 전체 파드 네트워크 대역. 다른 마스터 노드나 워커 노드에 살고 있는 파드로 패킷을 보낼 때 암호화된 wg0 통로를 거쳐서 전달한다.
    - 쿠버네티스 파드들이 나눠 가질 관례적인 기본 IP 대역이 10.244.0.0/16 이다.

- `Cilium CNI(Container Network Interface) 영역`
  
```
10.244.1.91 dev cilium_host proto kernel scope link 
10.244.1.0/24 via 10.244.1.91 dev cilium_host proto kernel src 10.244.1.91
10.244.0.0/24 via 10.244.1.91 dev cilium_host proto kernel src 10.244.1.91 mtu 8870 
10.244.2.0/24 via 10.244.1.91 dev cilium_host proto kernel src 10.244.1.91 mtu 8870 
10.244.3.0/24 via 10.244.1.91 dev cilium_host proto kernel src 10.244.1.91 mtu 8870
```

- 인터페이스 : cilium_host
- 이 노드의 호스트 파드 IP : 10.244.1.91
- CNI Cilium이 관리하는 라우팅 영역.
- 이 노드는 전체 파드 대역 10.244.0.0/16 중 10.244.1.0/24 라는 하위 대역을 배정받았다.
  - 내 노드에 떠 있는 파드들끼리 통신하거나, 다른 노드의 파드들과 데이터를 주고 받을 때 커널 수준에서 cilium_host 가상 문(10.244.1.91)을 거쳐서 패킷을 캡슐화하라는 규칙
  - mtu 8870 -> 점보 프레임 사용하여 터널링 오버헤드를 줄이고 eVPF 고속 통신을 하려고 Cilium이 셋팅해 둔 것.
