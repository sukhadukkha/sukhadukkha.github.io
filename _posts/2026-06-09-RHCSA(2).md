## Controlling access to Files with ACL

- ACL (Access Control List)
- ACL 왜 사용할까?
  - 기존 리눅스 권한 rwx 만으로는 세밀한 권한 제어가 어려울 때 사용

  - setfacl -Rm u:user1:rw /shared

## SELinux (Security Enhanced Linux)

- `DAC (Discretionary Access Control)` : 임의 접근 제어
  - 평소 사용하는 리눅스 권한 시스템 ex) -rwxr-x---
  - 파일 소유자가 권한을 마음대로 바꿀 수 있음.
- DAC의 문제
  - 예를 들어 Apache 웹 서버 해킹 -> root 권한 얻음 -> DAC에서는 root니까 뭐든 가능 -> /etc/shadow 읽기 등 가능
- `MAC (Mandatory Access Control)` : 강제 접근 제어
  - 사용자도 규칙을 마음대로 바꿀 수 없음
  - ex) Apache는 웹 서비스만 수행
  - httpd_t 프로세스는 httpd_sys_content_t 파일만 읽을 수 있음.
  - /home/jiho/index.html 777권한 이어도 MAC 입장에서는 user_home_t 파일이면 Apache는 못 읽음
- root 권한이 있더라도 허용된 행동만 하게 만드는 추가 보안 시스템
- 왜 필요한가? 
  - 해커가 Apache(httpd)를 해킹한 상황 -> Apache 프로세스가 root 권한을 얻어버리면 중요 파일에 접근 가능
  - SELinux가 있으면? Apache에 웹 파일 읽기, 웹 서비스 제공은 가능하지만 /etc/shadow 같은 중요 파일 읽기는 거부 (root 여도)

## Manage basic storage

- Local Storage
- `SAN (Storage Area Network)`
  - 일반 랜선이 아니라 오직 스토리지 통신만을 위한 초고속 광케이블과 SAN 스위치라는 전용 네트워크 인프라를 따로 구축하는 방식
  - 블록 단위 저장 : 서버에 연결하면 네트워크 드라이브가 아니라 리눅스에서 sdb,sdc 처럼 서버에 하드디스크를 꽂은 것 처럼 인식된다. 파일 시스템 생성도 가능하다.
  - 일반 네트워크 트래픽 간섭을 전혀 안받기에 속도가 압도적으로 빠르고, 지연시간 Latency가 거의 없다.
  - 매우 비싸고, 전문 기술 필요
- `NAS (Network Attached Storage)`
  - 대형 외장하드를 일반적인 이더넷(LAN선)에 물리적인 스위치를 통해 연결한 형태
  - 파일 단위 저장 : 여러대의 서버나 PC가 네트워크를 통해 하나의 대형 저장소에 동시에 접속해서 파일을 읽고 쓴다.
  - 가격이 저렴하고, 설정 간편하며 파일 공유가 쉽다.

- 1단계: 루트 파일 시스템을 읽기/쓰기로 변경하기 
  - 현재는 /sysroot가 read only로 마운트 되어있음
  - mount | grep sysroot 로 확인 가능 (ro)
  - /etc/shadow 를 수정해야 하는데 읽기 전용이면 수정 불가
  - mount -o remount,rw /sysroot
- 2단계: 실제 OS 환경으로 이동
  - chroot /sysroot -> 현재 루트 디렉터리 / -> /sysroot로 바뀜
  - 지금 initramfs 임시 환경에서 실제 OS 환경으로 이동
- 3단계: root 비밀번호 변경
  - passwd root
- 4단계: SELinux 재레이블 예약
  - 비밀번호 변경하면서 /etc/shadow 수정했는데, SELinux는 파일 컨텍스트가 이상하다고 생각할 수 있음
  - 다음 부팅 시 SELinux 컨텍스트 전부 재검사 하도록 예약
  - touch /.autorelabel -> 리눅스에는 특정 이름의 파일이 존재하면 동작하는 기능 존재
- 5단계: 종료
  - exit exit
